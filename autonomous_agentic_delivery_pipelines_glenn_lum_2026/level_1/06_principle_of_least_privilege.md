# L1-06 · Principle of Least Privilege

The instinct when wiring up an autonomous agent is to give it broad access and then narrow it later when something goes wrong. This is exactly backwards, and it is the single decision most likely to turn a useful pipeline into an incident report. Every credential you hand an agent is a capability that will be exercised — not necessarily in the way you imagined, not necessarily in response to the input you expected, and not necessarily only once. The question is not "what does the agent need to do its job?" The question is "what is the maximum damage this agent can do if it does the wrong thing at the worst possible moment?"

Saltzer and Schroeder formalised this in 1975, in a paper that has aged into one of the load-bearing texts of computer security. Their formulation: every program and every user should operate using the least set of privileges necessary to complete the job. The reasoning was about containment. If a process is compromised, or if it malfunctions, the damage is bounded by what it was allowed to do in the first place. This was true when the relevant threat was a buggy C program with a stack overflow. It is more true when the relevant actor is a probabilistic reasoner whose behaviour you cannot fully predict and whose mistakes have no obvious upper bound.

The shift from human operators to autonomous agents changes the calculation in a specific way. A human engineer with broad repository access is dangerous in principle but constrained in practice — they get tired, they read warnings, they hesitate before running commands they don't recognise. An agent has none of these brakes. It will execute the action it has decided on, at machine speed, across as many resources as its credentials reach. If those credentials extend to your production database, your cloud billing account, or your package registry, then a single confused reasoning step is enough to produce consequences that take a team weeks to unwind. The agent is not malicious. It is simply unconstrained, and unconstrained is the same as dangerous when the actor is non-deterministic.

The practical work of least privilege is the work of explicit enumeration. You list, for each agent or agent role, the exact set of repositories it can read, the exact set of files it can write, the exact commands it can execute, the exact APIs it can call, and the exact secrets it can resolve. Everything not on the list is denied. This sounds tedious, and it is, but the tedium is the point — the act of enumeration forces you to articulate what the agent is actually for, which is a piece of design work most teams skip. An agent that "helps with the codebase" cannot be scoped. An agent that "opens pull requests against the frontend repository, runs the test suite in a sandboxed container, and cannot push to main" can be scoped tightly because its purpose has been made concrete.

The vocabulary that grows around this work is worth knowing. Blast radius is the term for the maximum damage a component can inflict when it fails — and the design goal is to keep it small enough that a failure is a recoverable inconvenience rather than a recoverable-only-with-lawyers crisis. Permission scoping is the mechanism: tokens that grant access to one repository rather than the whole organisation, service accounts that can read one bucket rather than all buckets, runtime sandboxes that can write to a working directory but not to the filesystem at large. Fail-safe defaults is the discipline of starting from "denied" and adding grants explicitly, rather than starting from "allowed" and removing them when you notice problems. Each of these is straightforward to state and surprisingly hard to maintain, because the path of least resistance in any system is to reuse the broad credential that already works.

Two failure modes recur. The first is credential reuse: an agent inherits a developer's personal access token, or a CI runner's deployment key, because that token already has the permissions the agent happens to need. This collapses the audit trail (you can no longer tell whether an action was taken by the human, the CI system, or the agent) and it imports every other capability that credential carries. The second is scope creep: an agent that started with narrow permissions accumulates more over time as new use cases appear, each grant individually defensible, the aggregate indefensible. Both failure modes are organisational, not technical. The technical primitives for scoped tokens, role-based access, and sandboxed execution have existed for decades. What gets skipped is the discipline of treating each new permission as a deliberate expansion of blast radius.

The other half of least privilege is audit. A scoped capability is only as useful as your ability to verify, after the fact, what was done with it. Every privileged action an autonomous system takes needs to leave a record — what was done, when, by which agent, against which resource, in response to what trigger. Without this, you have no incident response. You have a system that produced an outcome, and a team trying to reconstruct causality from indirect evidence. The cost of building audit logging into an autonomous pipeline from the start is small. The cost of retrofitting it during an incident is essentially infinite, because the incident is already in progress and you are flying blind.

Treat permissions as a design surface, not as configuration to be filled in at the end. The question of what an agent can touch is the question of what an agent is — its scope of action is its identity in the system. Get this wrong and no amount of clever prompting, careful review, or downstream validation will compensate, because the agent will eventually do exactly what its credentials permit. Get it right and the rest of the pipeline becomes recoverable: every failure is contained, every action is attributable, and the worst case is bounded by what you decided in advance to accept.

## Level 2 candidates

**Permission scoping** — How you decompose "access to the system" into the exact set of capabilities a component needs, and deny everything else by default. Worth a deep dive because the mechanics — scoped tokens, narrow service accounts, capability-based APIs — vary across platforms and the wrong choice creates an audit trail that looks complete but isn't.

**Blast radius** — The practice of designing systems so that the maximum damage from any single component's failure is bounded and known in advance. Worth going deeper because the concept generalises beyond permissions into network segmentation, resource quotas, and rate limits, and because reasoning about it well is what distinguishes a pipeline that survives its first incident from one that doesn't.

**Token-based vs. role-based access** — The difference between a credential that carries its own permissions ("this token can do X") and one that derives them from an identity ("this role can do X, this user has this role"). Worth exploring because the tradeoff governs how access control scales across environments, teams, and ephemeral agent instances — and the wrong choice creates a credential management problem that grows superlinearly with team size.

**Audit logging** — What it takes to maintain a record of every privileged action that is complete, tamper-evident, and queryable after the fact. Worth a deep dive because the engineering decisions (what to log, where to store it, how to correlate across systems, how long to retain) determine whether you have post-incident analysis or post-incident speculation.

**Fail-safe defaults** — The design principle that systems should deny by default and require explicit grants to allow. Worth going deeper because it sounds trivial and is in fact one of the hardest disciplines to maintain in practice — every platform's defaults pull toward convenience, and the work of enforcing deny-by-default is continuous rather than one-time.
