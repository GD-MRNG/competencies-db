# L1-07 · DRY and the Encoding of Decisions

Most developers learn Don't Repeat Yourself as a rule about code: if you find yourself writing the same function twice, extract it. This is the shallow reading, and it has done real damage to how the principle is understood. Hunt and Thomas were not making a claim about line count. They were making a claim about knowledge — specifically, about where in a system a given decision is allowed to live. Read it that way and DRY stops being a refactoring habit and becomes an architectural commitment: every fact your system depends on has exactly one authoritative home, and everything else refers to it.

The reason this matters more in an agent pipeline than in a conventional codebase is that agents read everything you write. A human engineer scanning a repository ignores most of what they see; they know which file holds the real coding standards and treat the rest as historical noise. An agent does not have that filter. If your style guide says one thing in CONTRIBUTING.md, something slightly different in a workflow YAML comment, and a third variant in the system prompt you pass to the model, the agent will reconcile those sources by averaging, guessing, or — worse — picking whichever it saw most recently. The inconsistency is not absorbed by judgment; it propagates into output.

This is why duplication of instructions is not a code smell in an agent system. It is a correctness bug waiting to be triggered. The moment the same rule lives in two places, those two places will drift. Someone updates the prompt and forgets the documentation. Someone tightens the lint rule and forgets the prompt. Six weeks later you have an agent that confidently produces code conforming to a standard nobody on the team actually holds, because it learned that standard from a copy that nobody remembered was there. The system behaves inconsistently not because the model is unreliable but because you gave it inconsistent inputs and asked it to be consistent anyway.

The discipline DRY imposes is therefore a discipline about layering. Different kinds of knowledge belong in different kinds of artefacts, and mixing them is what produces the drift. Standards — the rules that describe what good output looks like — belong in configuration files that are version-controlled and read by every part of the pipeline that needs them. Reusable patterns — the workflows, templates, and structured instructions an agent invokes repeatedly — belong in something resembling a library: named, versioned, importable, and updated in one place. Enforcement — the gates that reject output that violates a standard — belongs in hooks and runners that execute deterministically, not in prose instructions hoping the agent will comply. Generation — the actual production of output — is where the agent does its work, and it should be the thinnest layer, drawing from the others rather than re-stating them.

When teams violate this layering, the symptoms are predictable. Enforcement gets smuggled into prompts ("please make sure you never commit secrets") where it is a suggestion rather than a guarantee. Standards get pasted into per-run instructions where they become snapshots that age out of sync with the canonical version. Reusable patterns get re-described inline in every workflow file because nobody built the skill library, so the same agent behaviour is specified five different ways in five different pipelines. Each of these is a local convenience that creates a global liability. The pipeline still runs. It just stops being trustworthy, in ways that are hard to attribute to any single decision.

The deeper consequence is what happens when you need to change something. In a DRY system, changing a standard is a one-line edit followed by a re-run. In a non-DRY system, changing a standard means finding every place it was copied, deciding whether each copy was a verbatim quote or an intentional variation, and updating each accordingly — without breaking the pipelines that depended on the old phrasing. This is the same cost asymmetry that defect amplification describes for bugs: a knowledge fact that takes seconds to update at its source takes hours to track down once it has been duplicated, and the longer the duplication has existed, the more the copies have quietly diverged.

There is a particular failure mode worth naming, because it is so common in agent setups: the workflow YAML as dumping ground. A workflow file is easy to edit, lives next to the thing it triggers, and feels like the natural place to put "just one more instruction." Over months, that file accumulates standards, enforcement attempts, prompt fragments, and ad-hoc patterns that should have been extracted long ago. By the time you have ten such workflows, you have ten subtly different definitions of what your agents are supposed to do, and no single place to read to find out what the truth is. The fix is not to clean up the YAML; it is to recognise that the YAML should have been calling into the right layer all along.

The skill this topic builds is the habit of asking, before you write any instruction, where this knowledge belongs. Is it a standard? Then it goes in configuration. Is it a reusable pattern? Then it goes in your skill library. Is it an enforceable constraint? Then it goes in a hook. Is it generation logic for a specific task? Then and only then does it belong in the prompt for that task. The question is mechanical once you internalise it, and the payoff is that your pipeline acquires the one property that makes any automated system maintainable: when something is wrong, there is exactly one place to fix it.

## Level 2 candidates

**Single source of truth** — Covers what it means in practice for a fact about a system to have exactly one authoritative representation, and how to identify when that property has been quietly broken. Worth deeper treatment because the failure modes (silent divergence, ambiguous authority, reconciliation by guesswork) have specific structural causes that are easier to prevent than to detect.

**Configuration drift** — Covers how environments and pipelines that begin identical accumulate small differences over time when configuration is not declaratively managed from a single source. Worth a deep dive because the dynamics are subtle — drift is rarely caused by a single bad change, but by the absence of a forcing function that keeps copies aligned — and the remediation patterns (immutable infrastructure, declarative configuration, drift detection) deserve treatment on their own terms.

**Abstraction as DRY** — Covers the distinction between extracting shared code to shorten files and extracting it so that a change to the underlying logic happens in exactly one place. Worth exploring because the wrong abstraction is often worse than the duplication it replaced, and the criteria for when extraction is genuinely warranted are not obvious from the surface symptom.

**Skill and template libraries** — Covers what it means to treat agent instructions as a versioned, named, importable artefact rather than inline prose copied between workflows. Worth deeper treatment because the engineering decisions (naming, versioning, dependency management, deprecation) mirror those of any shared code library, and getting them wrong recreates the duplication problem one level up.

**Documentation as code** — Covers what changes when documentation is subjected to the same versioning, review, and update discipline as source code, particularly when agents read that documentation as operational input. Worth a deep dive because the standard arguments for documentation discipline (drift, staleness, trust) acquire new weight when the consumer is an automated system that cannot distinguish authoritative from outdated content.
