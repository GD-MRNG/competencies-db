# L1-06 · Infrastructure as Code (Terraform) and GitOps

The console is a trap. Every cloud provider gives you a beautiful web UI with buttons and dropdowns, and the first time you provision a Lambda function or a database, clicking through it feels like the obvious move. It works. You see your resource appear. You move on. Six months later, something breaks, and nobody — including you — can answer the simple question: "what is actually deployed right now, and how did it get that way?" The console hides the most important thing about your infrastructure, which is its history. There is no diff. There is no commit message. There is no reviewer. The state of your production system exists only in the live cloud account, and the only way to inspect it is to click through hundreds of screens and hope you notice what's different from last week.

This is the problem Infrastructure as Code exists to solve, and it solves it by collapsing a category distinction that most developers carry around without realising it. You already treat application code as a versioned, reviewed, reproducible artifact. You would never edit production Python by SSHing into a server and changing a file. But you happily edit production infrastructure by logging into a console and changing a setting, because infrastructure feels like configuration rather than code. Terraform's central move is to insist that it isn't. Your VPC, your Lambda functions, your IAM roles, your database — all of it is code, and all of it deserves the same discipline.

Terraform itself is a declarative language. You don't write a script that says "create this resource, then create that one, then attach this thing to that thing." You write a description of the world you want — "there should be a Lambda function called `inference-handler` with these environment variables, attached to this IAM role, behind this API Gateway" — and Terraform figures out the difference between that description and what currently exists. If nothing exists, it creates everything. If something exists but drifted, it reconciles. If you delete a block from your code, Terraform deletes the resource. The declarative model matters because it makes your infrastructure code a description of truth rather than a sequence of historical actions. You can read a Terraform file and know what should be deployed. You cannot read a bash script of `aws` CLI commands and know the same thing — you'd have to mentally replay every command in order, including the ones that got commented out.

The thing that makes this work, and also the thing that bites everyone new to Terraform, is state. Terraform keeps a state file that maps the resources in your code to the actual resources in your cloud account — this `aws_lambda_function.inference` block corresponds to this real Lambda with this real ARN. Without state, Terraform would have no way to know whether to create a new resource or modify an existing one. With state, Terraform becomes powerful and slightly dangerous: the state file is the source of truth, and if it gets out of sync with reality (because someone clicked something in the console, or because two people ran Terraform at the same time), you get drift, conflicts, and occasionally accidental deletions. Storing state remotely with locking — typically in S3 with a DynamoDB lock table, or in Terraform Cloud — is not optional once more than one person touches the infrastructure.

GitOps is the operational discipline that wraps around this. The principle is simple: the git repository is the source of truth for infrastructure, and the only way to change production is to merge a pull request. No one runs `terraform apply` from their laptop against prod. Changes go through code review like any other code. CI runs `terraform plan` on the pull request and posts the diff as a comment, so reviewers can see exactly what resources will be created, modified, or destroyed before they approve. Once merged, CI runs `terraform apply` automatically. This gives you the things the console never could: an audit trail (every change is a commit with an author and a message), reproducibility (you can recreate the entire environment from the repo), and rollback (revert the commit, re-apply). When an auditor asks "who changed the IAM policy on the production database in March," you have an answer, and it took you ten seconds to find it.

The cultural shift this requires is real. Treating infrastructure as code means accepting that simple changes — changing a memory limit, opening a port — now require a pull request, a review, and a CI run instead of three console clicks. This feels slow at first. It is, in fact, slower for the first change. It is dramatically faster for the hundredth change, and it is the difference between recoverable and unrecoverable when something goes wrong at 2 AM. The team that can revert a commit and redeploy in five minutes is the team that sleeps. The team that has to remember what they clicked is the team that doesn't.

The skill this topic builds is the habit of treating your cloud environment as a software artifact. Once you internalise that, the rest of production engineering opens up: you can spin up identical dev, staging, and prod environments from the same modules; you can give junior engineers safe ways to propose changes; you can hand a new hire your repo and have them stand up a working copy of your stack on their first day. The Terraform syntax is a few weeks of learning. The mental shift — that infrastructure is code, that the repo is truth, that production changes go through review — is the actual competency, and it's the one that separates engineers who can run a system from engineers who can only build one.

## Level 2 candidates

**Terraform Basics and State Management** — Covers the HCL syntax, the resource/provider/module model, and the mechanics of state files including remote backends and state locking. Worth a deep dive because state is where Terraform stops being intuitive and starts being dangerous, and most production incidents involving Terraform are state incidents.

**Modules and Code Reuse** — Covers how to package reusable infrastructure patterns (a "standard VPC," a "standard Lambda-with-API-Gateway") as modules, and how to consume community modules from the registry. Worth going deeper because module design is the difference between a Terraform codebase that scales with your team and one that becomes a 5000-line copy-paste nightmare.

**Environment Separation (dev/staging/prod)** — Covers strategies for managing multiple environments — workspaces, separate directories, separate state files, separate accounts — and the trade-offs of each. Worth a dive because the choice you make here is hard to reverse and shapes how your team promotes changes for the life of the project.

**Secrets in Terraform** — Covers how to pass secrets into Terraform without committing them, how to reference Secrets Manager and Parameter Store from Terraform resources, and the specific footgun of secrets ending up in plaintext state files. Worth a dive because the obvious approaches are wrong in subtle ways and the consequences (a secret in a git-committed state file) are severe.

**GitOps and Change Control Pipelines** — Covers the CI/CD setup that makes GitOps real: `terraform plan` on PRs, `terraform apply` on merge, approval gates for production, and automated rollback strategies. Worth a dive because the pipeline is what makes the discipline enforceable rather than aspirational, and the patterns differ meaningfully between Terraform Cloud, Atlantis, and homegrown GitHub Actions setups.

**Drift Detection and Reconciliation** — Covers what happens when reality diverges from code (someone clicked in the console, a resource was modified out-of-band), how to detect it, and how to bring things back in sync without destroying production. Worth a dive because drift is inevitable in any real organisation and the techniques for handling it aren't obvious from the basics.
