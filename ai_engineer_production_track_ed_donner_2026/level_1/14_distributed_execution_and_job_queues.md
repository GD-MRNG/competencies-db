# L1-14 · Distributed Execution and Job Queues

The instinct most developers carry from their first web apps is that a request and a response are two ends of the same rope. The user pulls, the server pulls back, and the whole transaction completes inside a single HTTP round-trip. This works beautifully for fetching a row from a database or rendering a template. It falls apart the moment your backend is doing something that takes thirty seconds — and AI work almost always takes thirty seconds. The model thinks. A tool gets called. Another model thinks about the tool's output. Somewhere along the way a vector search runs. By the time you have an answer, the user's browser has timed out, the load balancer has given up, and your server thread has been held hostage for half a minute while doing essentially nothing.

Distributed execution is the architectural answer to this mismatch. The premise is simple: stop pretending that the request and the work are the same thing. When a user submits a job, your web server's only responsibility is to write down "this job needs doing" somewhere durable and immediately return an acknowledgement. The actual work happens elsewhere, on a separate fleet of machines whose entire purpose is grinding through that backlog. The user finds out the result through a different channel — by polling, by webhook, by a websocket push — whenever the work actually finishes. The web server is free in milliseconds. The worker takes as long as it takes. Nobody is blocked waiting on anyone.

The mental model to hold in your head has four moving parts. There is a producer (your web server, accepting requests from users). There is a queue (a durable buffer that holds pending work — SQS, Redis, RabbitMQ, whatever your stack favours). There is a worker pool (a set of processes, containers, or serverless functions that pull jobs off the queue and execute them). And there is a result store (usually your database, sometimes a cache) where workers write their output and from which the frontend eventually reads it. The queue is the load-bearing piece. It decouples the rate at which work arrives from the rate at which work gets done, which is the entire point. If a thousand users submit jobs in a single second, the queue absorbs them and the workers chew through the backlog at whatever pace they can sustain. Nothing crashes. Nothing times out. The system degrades into slowness rather than failure, which is the only graceful kind of degradation there is.

Once you internalise this shape, a lot of operational concerns rearrange themselves. Scaling becomes about queue depth: if the backlog grows, spin up more workers; if it shrinks, scale down. You can run workers as Lambda functions if the jobs are short and bursty, or as long-lived containers (ECS, App Runner, Kubernetes pods) if they are heavy, stateful, or benefit from warm caches. The choice matters less than the fact that workers are now an independent dimension of your architecture, scalable on their own axis, deployable on their own cadence. Your web server can stay small and cheap because it does almost no real work; your worker fleet can be large and expensive because that is where the actual computation lives.

The part that catches people out is what happens when things go wrong, because distributed systems give you a new and unfamiliar failure mode: the half-done job. A worker pulls a job off the queue, starts processing, and then crashes — maybe the container is killed, maybe the network drops, maybe an LLM call hangs forever. Most queues handle this with at-least-once delivery: if a worker doesn't acknowledge completion within some timeout, the job goes back on the queue and another worker picks it up. This is the right default, but it forces a discipline on you: your jobs must be idempotent. Running the same job twice must produce the same result as running it once. If your job charges a credit card, you cannot afford to retry naively. You need request IDs, deduplication tables, or transactional outboxes — some mechanism that makes "did I already do this?" a question with a definite answer. The jobs that survive longest in production are the ones designed assuming they will be retried at the worst possible moment.

The result-delivery half of the system has its own design choices. Polling is the simplest: the frontend asks "is job 1234 done yet?" every few seconds until the answer is yes. It works everywhere, requires no special infrastructure, and wastes bandwidth on the negative answers. Webhooks invert the relationship — the worker calls back into the requester's system when the job completes — which is efficient but only works when the requester has a publicly reachable endpoint, so it tends to be a business-to-business pattern. Websockets and server-sent events give you a persistent channel from server to browser, which is the right answer for interactive UIs where the user is sitting there waiting. Most real systems use a mix: SSE for the user who is actively watching, webhooks for the integration partner, polling as the universal fallback.

The skill this topic builds is the ability to look at a synchronous request flow and see where the seams are. Where is the work that could be deferred? Where is the user waiting on something they don't actually need to wait on? Where would your system survive if you replaced an in-process function call with a queue and a worker? Once you can see those seams, you can scale past the limits of a single machine, survive bursts you didn't plan for, and run jobs that take minutes or hours without anyone noticing. The web tier becomes a thin acknowledgement layer. The real work lives behind the queue, where it belongs.

**Job Queues and Message Passing** — Covers the semantic differences between queue technologies (SQS, Redis streams, RabbitMQ, Kafka), including FIFO vs. best-effort ordering, priority lanes, visibility timeouts, and dead-letter queues for poison messages. Worth a deep dive because the choice of queue locks in a set of guarantees you'll either rely on or fight against for the life of the system.

**Worker Pools and Autoscaling on Queue Depth** — Covers how to size a worker fleet, how to scale based on backlog rather than CPU, and how to balance warm-pool cost against cold-start latency. Worth deeper exploration because most teams either over-provision and burn money or under-provision and watch queues grow unbounded during traffic spikes.

**Idempotency and At-Least-Once Delivery** — Covers the design patterns that make jobs safe to retry: idempotency keys, deduplication tables, transactional outboxes, and the distinction between retry-safe and retry-dangerous side effects. Worth deeper exploration because this is where distributed systems go from "works in testing" to "doesn't double-charge customers in production."

**Result Delivery: Polling, Webhooks, and Push Channels** — Covers the trade-offs between polling, webhook callbacks, SSE, and websockets for delivering completed work back to the caller. Worth deeper exploration because the right choice depends on whether your consumer is a browser, a mobile app, or another backend, and the wrong choice produces either wasted bandwidth or unreachable callbacks.

**Lambda vs. Container Workers** — Covers when to run workers as serverless functions versus long-lived containers, including cold-start implications, maximum execution time, statefulness, and cost curves at different volumes. Worth deeper exploration because AI workloads sit awkwardly on the boundary — too long for many serverless platforms, too bursty for a static container fleet — and the right answer is often a mix.

**Backpressure and Queue Overflow** — Covers what happens when work arrives faster than workers can process it indefinitely, including bounded queues, rejection strategies, and shedding low-priority work to protect the system. Worth deeper exploration because most engineers design for the happy path where the queue drains; the interesting failure mode is the queue that never drains.
