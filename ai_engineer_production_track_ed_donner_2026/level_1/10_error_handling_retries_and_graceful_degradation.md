# L1-10 · Error Handling, Retries, and Graceful Degradation

The mental shift you need to make for production AI is this: failure is not an exception, it is a routine event you design around. In a normal web application, a 500 error is a bug to be fixed. In a system built on LLM APIs, vector stores, embedding services, and tool calls, errors are a constant background hum. Providers have outages. Rate limits trip. Requests time out. A model that returned valid JSON yesterday returns prose today. If your reflex is to treat each failure as something to debug after the fact, you will spend your nights paging through logs and your days apologising to users. The reflex you want instead is to assume every external call will fail at some point, and ask: what does the system do when it does?

The first layer of defence is retries, and the version that works is retries with exponential backoff. The naive version — retry immediately, three times, then give up — is worse than no retries at all, because it amplifies load on a service that is already struggling. Exponential backoff means each retry waits longer than the last (one second, two seconds, four seconds), giving the upstream service room to recover and preventing your fleet from turning into an accidental DDoS. But retries are only safe for a narrow class of failures: transient ones. A timeout, a 429 rate limit, a 503 from the provider — retry. A 400 bad request, a malformed schema, an auth failure — do not retry, because no amount of waiting will make the request valid. Knowing which is which is the difference between a resilient system and one that burns money retrying requests that will never succeed.

Retries solve the problem of brief, transient failures. They do not solve the problem of a service that is actually down. If your LLM provider has a regional outage and you keep retrying every request with exponential backoff, every user session ties up resources for minutes, your queues back up, and the whole system grinds to a halt waiting for something that is not coming back. This is where circuit breakers come in. A circuit breaker tracks the failure rate of calls to a downstream service. When failures cross a threshold, the breaker "opens" and subsequent calls fail immediately without even attempting the network round trip. After a cooldown, it tentatively allows a request through to test if the service has recovered. The point is not to hide the failure — it is to fail fast and cheaply, so the rest of your system stays responsive while one dependency is sick.

Failing fast is necessary, but it is not sufficient. A request that fails fast is still a failed request from the user's perspective. The third layer is graceful degradation, and this is where the design thinking gets interesting. The premise is that your system has features, not just dependencies, and many features have a reduced-but-still-useful mode. If your vector database is slow or down, you can skip the RAG step and answer from the base model alone — less informed, but still an answer. If the embedding service is failing, you can fall back to keyword search instead of semantic search. If the frontier model is unreachable, you can route to a smaller, cheaper backup model. The user gets a degraded but functional experience instead of an error page. This requires you to think about your system as a stack of capabilities with explicit fallback paths, not a brittle chain where any broken link kills the whole request.

Two adjacent concepts make all of this safe in practice. The first is timeouts. Every external call needs an explicit timeout, because the default in most libraries is either "wait forever" or "wait several minutes," and a hung request consumes a worker, a database connection, and eventually all your capacity. An LLM call that should complete in ten seconds should be cancelled at thirty, not left hanging at five minutes while the rest of your system suffocates. The second is idempotency. If you retry a request, you need to guarantee that retrying does not double-process it — charging a user twice, sending two emails, writing two database rows. The standard pattern is to attach a unique request ID to every operation and have the receiving service deduplicate based on it. Without idempotency, retries become a footgun: every transient blip in the network risks doing the work twice.

The deeper shift here is accepting that perfect availability is a fantasy. Real systems have error budgets — a target like "99.9% availability" implicitly says "0.1% failure is acceptable." That is not laziness, it is honesty. Chasing the last fraction of a percent costs disproportionately more than the previous one, and the engineering effort is usually better spent making failures graceful than making them impossible. When you measure your actual failure rate against your target, you know whether to invest in reliability work or move on to features. Without that target, every outage feels like a crisis and every retry mechanism feels insufficient.

The skill this topic builds is a particular kind of paranoia, applied with discipline. You learn to look at every external call in your codebase and ask four questions: what happens if this times out, what happens if this returns an error, what happens if this returns success but with bad data, and what happens if this is retried twice. If you cannot answer all four for every call, your system has bugs you have not seen yet. The users will find them for you, usually at 3 AM. Design for failure now, or debug it later under pressure — those are the only two options.

## Level 2 candidates

**Retry Logic and Exponential Backoff** — Covers when to retry, how to implement backoff with jitter, and how to classify errors as retryable vs. terminal. Worth a deep dive because the naive implementation is actively harmful and the correct one has surprising subtleties (jitter to prevent thundering herds, max retry budgets, distinguishing 429 from 5xx).

**Circuit Breakers and Fallbacks** — Covers the state machine of a circuit breaker (closed, open, half-open), failure thresholds, and how to design fallback responses. Worth deeper exploration because the tuning of thresholds and cooldowns is where most implementations get it wrong, and the design of meaningful fallbacks is genuinely creative work that varies by feature.

**Idempotency and Duplicate Prevention** — Covers idempotency keys, deduplication storage, and the patterns for making non-idempotent operations safe to retry (writes, payments, external API calls). Worth a deep dive because this is conceptually simple but operationally tricky — where you store the idempotency keys, how long you keep them, and how you handle concurrent retries are all nontrivial.

**Timeouts and Resource Limits** — Covers setting timeouts at every layer (HTTP client, database, LLM call), per-function memory and CPU limits, and how cascading timeouts should be configured. Worth deeper coverage because the right timeout depends on the call's position in a chain, and getting this wrong causes the most insidious production failures (gradual resource exhaustion rather than loud crashes).

**Error Budgets and Acceptable Failure Rates** — Covers SLO/SLI definitions, error budget accounting, and how reliability targets should drive engineering priorities. Worth its own treatment because it is as much an organisational and product concept as a technical one, and learning to think in budgets rather than absolutes changes how you allocate engineering time.
