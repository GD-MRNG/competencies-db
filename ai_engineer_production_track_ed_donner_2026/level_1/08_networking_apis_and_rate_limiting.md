# L1-08 · Networking, APIs, and Rate Limiting

Networking is the most common blind spot for developers moving into operational work, and it is the most consequential one. When you think about your code, you think in terms of function calls, objects, and data structures. When a request fails in production, however, the failure is almost never inside the code itself; it is almost always in the path the request takes to reach the code, or in the path the response takes to return from it. Your FastAPI handler is fine. Your model call is fine. But the browser can't reach the backend, or the backend can't reach the database, or the gateway is throttling you, or a single user has discovered they can drain your monthly OpenAI budget in an afternoon. These are networking problems, and they are what separate a prototype from a production system.

The mental model to start with is this: every component in your system lives somewhere on a network, and every connection between components is either explicitly allowed or implicitly forbidden. A Virtual Private Cloud (VPC) is the boundary you draw around your infrastructure to make "implicitly forbidden" the default. Your database goes in a private subnet — no route to the internet, no inbound traffic from anywhere except the specific services you've authorised. Your backend lives in a subnet that can talk to the database and can be reached by your load balancer. Your load balancer is the only thing the public internet can see. This is not paranoia; it is the only sane default. A database with a public IP and a leaked password is a breach waiting for a port scanner to find it, and port scanners find everything.

Once you've decided what is reachable, you need to decide how. HTTPS is non-negotiable for anything a user touches — not just because browsers will refuse to call non-HTTPS endpoints from an HTTPS frontend, but because anything sent in plaintext is anything an intermediary can read or modify. TLS certificates used to be a nightmare; today, services like AWS Certificate Manager or Let's Encrypt issue and rotate them for free, and your load balancer terminates TLS so your application never has to think about it. The cost of getting this right is fifteen minutes of configuration. The cost of getting it wrong is a credential leaked over an airport Wi-Fi network.

In front of your backend sits an API Gateway, and this is where a lot of cross-cutting concerns get solved cheaply. An API Gateway routes incoming requests to the right backend based on path or method (`/v1/chat` goes to your chat container, `/v1/embed` goes to your embeddings Lambda), enforces authentication, applies rate limits, and gives you a single stable URL even as you reshuffle the services behind it. Without a gateway, every backend service has to reimplement the same auth and rate-limiting logic, and your frontend has to know about every backend's URL. With a gateway, the frontend talks to one place and the gateway handles the dispatching. This is the single most useful piece of infrastructure you can put in front of an AI system, because it gives you a chokepoint where you can observe, throttle, and protect everything downstream.

Rate limiting deserves its own paragraph because in AI systems it is not a nice-to-have, it is survival. An LLM call costs real money, and your provider has its own rate limits that, if you hit them, will cause every user to see errors at once. You need to limit requests per user, per API key, and per IP — not because most users are malicious, but because one buggy script in a customer's codebase can issue a thousand requests a second and either bankrupt you or take down the service for everyone else. The gateway is the natural place to enforce this, because it sees every request before it reaches anything expensive. Define limits that match your unit economics: if a user is on a $20/month plan and each request costs you $0.02, somewhere around a thousand requests a day is where you should be slowing them down. The gateway returns a 429 and your backend never knew the request existed.

API keys are how you identify who is making each request, and they are also how you bill, throttle, and revoke. Issue a unique key per customer (or per environment, per integration — never one shared key for everyone), give each key a scope (read-only, embeddings-only, full access), and treat the key as a database row that you can disable instantly without touching code. When a customer's key leaks — and it will — you revoke that one key, you do not rotate a global secret. This is the same Principle of Least Privilege you applied to IAM roles, applied now to your own users.

Finally, your API itself is a contract. The moment a frontend ships against `/v1/chat` returning `{message: string}`, you cannot change that response shape without breaking every client that depends on it. This is why production APIs are versioned (`/v1/`, `/v2/`) and why breaking changes go in a new version while the old one keeps working until you can prove no one uses it. The corollary for browser-based frontends is CORS — Cross-Origin Resource Sharing — which is the mechanism by which your backend on `api.example.com` declares that the frontend on `app.example.com` is allowed to call it. Get the CORS headers wrong and the browser silently refuses every request, and you spend an afternoon convinced your backend is broken when actually it's responding fine and the browser is throwing it away.

The skill this topic builds is the ability to think about your system as a graph of trust boundaries rather than as a pile of code. Every arrow between two nodes is a decision: who is allowed to traverse it, over what protocol, at what rate, with what credentials, and what happens when it fails. Once you can draw that graph for your own system and defend every edge, you have stopped being someone who writes code and started being someone who runs services.

## Level 2 candidates

**VPCs and Network Isolation** — Covers private vs. public subnets, security groups, NAT gateways, and the practical patterns for placing databases, backends, and load balancers inside an AWS-style VPC. Worth a deep dive because the default AWS networking concepts (subnets, route tables, security groups vs. NACLs) are genuinely confusing the first time and getting them wrong creates security holes that are invisible until exploited.

**HTTPS and TLS in Practice** — Covers certificate issuance and rotation, TLS termination at the load balancer vs. application, and how custom domains map to cloud services. Worth deeper treatment because the conceptual model of certificate authorities, chains of trust, and SNI is non-obvious, and the failure modes (expired certs, mixed-content warnings) are common production incidents.

**API Gateway Patterns** — Covers routing rules, request/response transformation, authentication integration, and the trade-offs between API Gateway, Application Load Balancer, and reverse proxies like nginx or Traefik. Worth a deep dive because the choice of gateway shapes how you implement auth, rate limiting, and versioning across the entire system.

**Rate Limiting Strategies** — Covers token bucket vs. leaky bucket algorithms, per-user vs. per-IP vs. per-key limits, distributed rate limiting with Redis, and how to communicate limits to clients via headers like `Retry-After`. Worth deeper treatment because naive rate limiting (e.g., per-instance counters behind a load balancer) silently fails to actually limit anything, and the right design depends on your traffic shape.

**API Versioning and Backward Compatibility** — Covers URL versioning vs. header versioning, deprecation cycles, and strategies for evolving response schemas without breaking clients. Worth a deep dive because the cost of getting versioning wrong compounds — every breaking change requires coordinating with every consumer — and the discipline is easier to adopt early than retrofit.

**CORS and Browser Security** — Covers preflight requests, allowed origins, credentials handling, and the common mistakes that lead to silent request failures. Worth deeper treatment because CORS failures present as cryptic browser console errors that don't reach your backend logs, and most developers learn it through painful trial and error rather than understanding the underlying model.
