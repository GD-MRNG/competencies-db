# L1-04 · Async Backends and Streaming

Most backend code you've written probably didn't care about concurrency. A request came in, your handler ran, a response went out, and the framework underneath quietly managed threads on your behalf. You could afford not to think about it because the work inside each handler was fast — a database query here, a cache lookup there, maybe a few milliseconds of computation. The thread that picked up your request was free again almost immediately, ready for the next one. Concurrency was somebody else's problem.

LLMs break this comfortable arrangement. A single model call can block for ten, twenty, sometimes thirty seconds while the provider streams tokens back to you. From your server's perspective, that's an eternity of doing nothing — just waiting on a socket for bytes to arrive. If the thread handling that request is held hostage for the full duration, the math gets ugly fast. A typical server might have a worker pool of a few dozen threads. Drop a hundred concurrent users on it, each waiting on a slow model call, and you've exhausted the pool. New requests don't get rejected cleanly; they queue, time out, and the whole system grinds. Your "AI app" is down, and the bottleneck isn't the model — it's the way your backend waits.

The fix is to stop waiting like that. Asynchronous backends, built on Python's `async`/`await` syntax and an event loop underneath, treat I/O waits as opportunities to do something else. When your handler reaches an `await` on a slow network call, it doesn't hold the thread hostage — it yields control back to the event loop, which immediately picks up another request to make progress on. When the bytes finally arrive from the model provider, the event loop resumes your original handler exactly where it left off. A single thread can juggle hundreds or thousands of in-flight requests this way, because at any given moment, almost all of them are just waiting on the network. The CPU is barely working. That's the trick: async doesn't make anything faster, it just stops you from blocking on nothing.

This is why FastAPI (and its peers in other languages — Node, Go's goroutines, the same idea wearing different clothes) has become the default choice for AI backends. It's built around the assumption that your handlers will spend most of their time awaiting slow upstreams. You write `async def` route handlers, you `await` your model client, and the framework wires the rest together. The mental shift is small but real: you have to remember that any blocking call inside an async handler — a synchronous HTTP library, a CPU-bound loop, a `time.sleep` — defeats the whole mechanism by hogging the event loop. One careless `requests.get` in an async route and you're back to thread-starvation territory, just with more confusing symptoms.

Async also unlocks the second half of this topic: streaming. Once your backend can hold a connection open without burning a thread, you can do something the model providers already support natively — push tokens to the user as they're generated, rather than waiting for the complete response. The wall-clock latency hasn't changed; the model still takes twenty seconds to finish. But the user sees the first word in 300 milliseconds, and the experience is transformed. This is mostly perception, and perception is most of what users care about. The two common mechanisms are Server-Sent Events (a simple one-way stream over HTTP, well-suited to chat-style interfaces) and WebSockets (bidirectional, heavier, useful when the client needs to interrupt or send messages mid-stream). SSE is usually enough. The cost you pay is frontend complexity: you're no longer rendering a finished JSON blob, you're appending chunks as they arrive, handling partial states, and gracefully recovering when the connection drops mid-stream.

There's a deeper lesson hiding underneath all this, which is that async exposes failure modes that synchronous code lets you ignore. When a single thread is managing a thousand concurrent requests, the question of what happens when the model provider gets slow becomes urgent. Do you keep accepting new requests forever? At some point you run out of memory tracking them. This is where backpressure and rate limiting stop being theoretical concerns and become survival mechanisms — you need to cap how many requests you'll hold in flight, reject excess load with a clear error rather than silently degrading, and set timeouts so a hung upstream doesn't accumulate zombie requests forever. Async makes high concurrency cheap, which means it also makes runaway concurrency cheap, and you have to design against it deliberately.

The skill this topic builds, then, isn't really "use FastAPI" or "write `async def`." It's the ability to reason about your backend as a system that spends most of its time waiting on something slower than itself. Once you internalize that, the architectural decisions follow naturally — async by default, stream what you can, set timeouts everywhere, cap your concurrency, and never put a blocking call in an async handler. For AI workloads, none of this is optional. You can ignore it on a demo. You cannot ignore it the first time real traffic hits.

**Server-Sent Events vs. WebSockets for Streaming** — The concrete protocols for pushing tokens to a frontend, their trade-offs (SSE is simpler and one-way, WebSockets are bidirectional and heavier), and how to implement reconnection and partial-state handling on the client. Worth a Level 2 because the protocol choice and the frontend complexity are real engineering decisions, not implementation details.

**Rate Limiting, Backpressure, and Timeouts** — How to protect an async backend from the failure modes that async itself enables: unbounded in-flight requests, slow upstreams holding connections forever, single users monopolizing capacity. Worth its own deep dive because the patterns (token buckets, queue caps, deadline propagation, graceful 429s) are subtle and the consequences of getting them wrong are outages.

**Blocking Calls in Async Code: Diagnosis and Escape Hatches** — How to identify when a sync library is poisoning your event loop, and the standard mitigations (`run_in_executor`, thread pools for CPU-bound work, process pools when you actually need parallelism). Worth its own treatment because in practice this is the single most common bug in real async codebases, and the fixes require understanding the boundary between async and threaded execution.
