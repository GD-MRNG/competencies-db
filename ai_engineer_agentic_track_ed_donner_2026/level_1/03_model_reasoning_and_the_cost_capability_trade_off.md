# L1-03 · Model Reasoning and the Cost-Capability Trade-off

The instinct most developers carry over from single-shot generation is that model choice is a price-per-token negotiation. You pick the cheapest model that produces acceptable output for your task, and you save money. In agentic systems, this instinct will quietly destroy you. The model is no longer producing one answer for you to inspect; it is making a sequence of decisions, each of which feeds the next, and each of which compounds. A model that is 90% reliable at picking the right tool will, over a ten-step loop, get the whole sequence right less than 35% of the time. A model that is 99% reliable will get it right 90% of the time. The gap between "almost good enough" and "good enough" is the gap between a system that works and a system that burns tokens going in circles.

What you are actually buying when you pay for a frontier model is reasoning effort — the model's capacity to hold a goal in mind, evaluate the current state against that goal, scan a list of available tools, pick the right one, and format the call correctly. This sounds trivial when you describe it. It is not. Each of those sub-steps is a place where smaller models fail in ways that look superficially like the right behavior but break the loop. A small model will confidently call a tool that doesn't exist. It will pass arguments in the wrong shape. It will choose the third tool when the second was obviously correct, because the third had a more descriptive name. None of these failures look like the model "not knowing the answer" — they look like the model executing competently in the wrong direction.

The cost structure follows from this. A frontier model might cost ten or twenty times more per token than a small one, and the naive reaction is to reach for the cheap option and accept some failures. But the failure mode of a cheap model in an agentic loop is not "wrong answer once." It is "wrong tool, observation makes no sense, model tries to recover, picks another wrong tool, loops until the iteration budget runs out." You end up paying for fifty calls to the cheap model to get a result that one call to the expensive model would have produced — and getting a worse result anyway. Cheap is broken, in this context, in a way that cheap is not broken for a one-shot summarization task.

This shifts how you evaluate models for agentic work. Standard benchmarks measure things like reasoning, knowledge, and instruction following on isolated prompts. What you actually care about is tool selection accuracy across a list of plausible options, schema adherence under pressure, and self-correction — whether the model, given a tool result that clearly indicates failure, recognizes the failure and changes approach rather than doubling down. These are different capabilities than benchmark scores measure, and they vary considerably between models that look similar on paper. The only reliable way to know is to test the specific models you're considering against the specific tool surface your agent will use.

The escape hatch from the cost problem is not to use a cheaper model for the whole loop; it is to decompose the loop so that different models handle different jobs. The planner — the agent that decides what to do next and which tools to call — needs to be a frontier model, because that is where the hard reasoning lives. The workers — agents that execute narrow, well-defined sub-tasks like "extract these fields from this document" or "summarize this passage" — can be smaller, cheaper models, because their job is bounded and their failure modes are easier to detect. This is the architecture that lets you scale agentic systems economically: spend the money where the reasoning is, save the money where the execution is. It is also the bridge into multi-agent design, which is a topic in its own right.

Even with the right model, you need hard limits. An agentic loop has no native concept of "I should stop now." It will happily call tools until it hits an external constraint. So you set them explicitly: a maximum number of iterations, a token budget for the full loop, a wall-clock timeout, an explicit exit condition the model is instructed to recognize. These are not optimizations; they are safety rails. Without them, a single misconfigured agent can burn through a month's API budget over a weekend. The frontier models are better at recognizing dead ends and stopping voluntarily, which is another reason the cost calculus is not as simple as price-per-token, but you still need the rails.

One note about model names: the field moves quickly enough that any specific model I could name as "frontier" today will likely be displaced within a release cycle or two. Treat any reference to a specific model — in this course, in blog posts, in framework documentation — as illustrative rather than authoritative. The skill is knowing what capabilities to test for, not memorizing which lab is currently winning. Check current benchmarks, but more importantly, run your own evaluation against your own tool surface. The model that's best at general reasoning is not always the model that's best at calling your specific tools, and the only way to find out is to measure.

The practical takeaway is that model selection in agentic systems is not a cost decision; it is a reliability decision with cost implications. Pick the model that can actually drive your loop end-to-end, set hard limits on what it can spend, and decompose the work so you're only paying frontier prices where frontier reasoning is required. The teams who get this wrong spend their first month building agents that look like they work in demos and fall apart in production. The teams who get it right spend a bit more per call and ship systems that finish their loops.

Worth its own treatment because self-correction is where the real reliability gap between models shows up, and it's largely invisible until your agent is already failing in production.

**Token Budgets and Iteration Limits** — The practical mechanics of putting hard limits on agentic loops: where to enforce them, what to do when they trip, and how to surface partial results gracefully. Worth deeper coverage because the default failure mode without limits is catastrophic and the implementation details (per-loop vs. per-agent budgets, soft vs. hard caps) matter more than they appear.
