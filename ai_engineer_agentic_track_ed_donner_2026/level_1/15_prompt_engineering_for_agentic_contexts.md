# L1-15 · Prompt Engineering for Agentic Contexts

The prompt engineering you learned for single-shot generation is the wrong mental model for agents. In a one-shot prompt, an ambiguity in your instructions produces one ambiguous answer; you read it, you adjust, you move on. In an agentic loop, that same ambiguity is consulted ten, fifty, a hundred times — once per iteration — and each consultation compounds the drift. The agent is not reading your System Message once and producing an output. It is re-reading it on every turn, using it to interpret tool results, to decide whether to continue, to choose what to do next. Your prompt is no longer a question. It is a constitution.

This changes what good prompting looks like. A single-shot prompt optimises for one trajectory: get the model to produce the answer you want, this time. An agentic System Message optimises for stability across many trajectories: the agent should behave the same way whether it's on iteration two or iteration thirty, whether the last tool call succeeded or returned garbage, whether the user's request landed in a well-trodden path or an edge case. The metric you care about is not "did this run produce a good output" but "does this agent's behaviour stay coherent under variation." Small ambiguities you would never notice in a chat prompt — a verb that could mean two things, a missing constraint, an implicit assumption about ordering — become the source of the loops, the wrong tool selections, and the inexplicable failures you'll spend days debugging.

The first thing this demands is explicit goal definition. In a chat prompt, "help the user analyse their sales data" works because the user is there to clarify in the next message. In an agentic context, there is no next message from a human — there's just the agent looking at a tool result and asking itself, "am I done?" If your goal statement doesn't have a recognisable success condition, the agent has no basis for that judgement, and you get either premature termination (it stops at the first plausible-looking output) or runaway iteration (it keeps calling tools because nothing tells it the goal is met). A good agentic goal reads more like an acceptance criterion than a task description: what does "done" look like, what does the final output contain, what must be true for the agent to stop.

The second is tool-use discipline. The model has a list of tools, and the System Message is where you teach it the policy for using them — not just what each tool does (the tool description handles that) but when to reach for one versus another, what order makes sense, and what to do with results that look unexpected. Without this layer, the model falls back on its training-data instincts about which tool sounds most relevant to the user's last message, which is exactly the heuristic that breaks down in multi-step problems. You want to be explicit: "use the search tool first to identify candidate records; only call the detail-fetch tool on records that match these criteria; never call the write tool without first reading current state." This kind of constraint reads like over-specification until you watch an under-constrained agent burn a hundred thousand tokens exploring the wrong subtree.

The third is constraints and safety boundaries — the negative space. Telling the agent what it can do leaves enormous room for what it shouldn't. In a one-shot prompt, the worst case of an unspecified constraint is a slightly off output. In an agentic loop with real tools, the worst case is the agent helpfully sending an email, deleting a record, or making a payment because nothing in the System Message told it not to. Explicit prohibitions, escalation criteria ("if the requested action affects more than 100 records, return a confirmation request rather than executing"), and refusal conditions belong in the System Message, not implicit in your hope that the model has good judgement.

The fourth is recovery. Agents get stuck. They call a tool, it returns an error or an empty result or something they didn't expect, and a poorly-prompted agent will simply call the same tool again with slightly different arguments, then again, then again, until you cut it off. Your System Message has to anticipate this and give the model a script: if a tool fails twice in a row, change strategy; if you cannot make progress after N steps, summarise what you've tried and return; if the goal becomes unreachable, say so explicitly rather than fabricating an answer. This is the single highest-leverage section of an agentic prompt, because the difference between an agent that recovers gracefully and one that doesn't is not the model's capability — it's whether you told it that recovery was part of the job.

Finally, treat the System Message as something you iterate on with evidence, not something you write once. Because agentic behaviour is stochastic and path-dependent, you cannot validate a prompt by running it once and reading the output. You validate it by running it many times across varied inputs and watching the distribution of behaviours: does it loop on certain queries, does it pick the wrong tool when the user phrases something a particular way, does it terminate early on a class of inputs you care about. Each pattern you find sends you back to the System Message to tighten the relevant clause. The prompt converges over runs, not over re-reads.

The shift from one-shot prompting to agentic prompting is the shift from writing instructions to writing policy. A good agentic System Message is closer in spirit to a job description than to a question: it tells the agent what its role is, what success looks like, what it's allowed to do, what it's not, and what to do when reality doesn't match the plan. Get this layer right and most of the other failure modes in agentic systems — infinite loops, wrong tool choices, hallucinated results, runaway costs — get easier to fix because you have a stable substrate to fix them against. Get it wrong and no amount of better models, better tools, or better frameworks will compensate.

## Level 2 candidates

**Goal Specification and Success Criteria** — How to write a goal statement that the agent can use as a termination signal, not just a task description, including the difference between vague objectives and acceptance criteria. Worth deeper treatment because the failure modes (premature stop, runaway loops) are subtle and the techniques for diagnosing which one your prompt is causing aren't obvious.

**Tool-Specific Instructions** — How to layer policy on top of tool descriptions: when to use each tool, in what order, what to do with unexpected results. Worth its own treatment because there's a real craft to writing these without over-specifying (and locking the agent out of legitimate alternative paths) or under-specifying (and watching it flail).

**Constraints and Safety Boundaries** — The negative space of the System Message: what the agent is forbidden from doing, escalation criteria, refusal conditions, and how these interact with tool-level guardrails. Worth going deeper because the consequences of getting this wrong are concrete and irreversible in production, and the patterns for expressing constraints reliably are non-trivial.

**Iteration and Stability** — The methodology of refining a System Message against distributions of behaviour rather than single outputs: how to run varied inputs, what to look for, how to know when a prompt has converged. Worth a deep dive because most teams don't have a process for this and end up shipping prompts that work on the three examples they tested.

**Handling Loops and Stuck States** — Explicit recovery instructions: how to tell the agent to detect repetition, change strategy, summarise progress, or escalate. Worth deeper coverage because this is the single highest-leverage section of an agentic prompt and the specific phrasings and structures that work well are learnable but not obvious.
