# L1-08 · Tool Definition and Schema Enforcement

The phrase "the agent called a tool" hides a small lie. The agent did not call anything. The model generated a string of text that, if you squint, looks like a function call — a name, some arguments, maybe wrapped in JSON. Your code is what actually called the tool. The model just produced characters. Everything that makes an agent feel like it's reaching out and touching the world depends on the bridge between that generated text and the function you run on its behalf, and that bridge is narrower and more fragile than most people expect.

This is why tool definition is the unglamorous load-bearing wall of every agentic system. If the model generates `{"city": "Paris"}` when your function expects `{"location": "Paris"}`, the call fails. If it emits a trailing comma, your JSON parser throws. If it invents a parameter that doesn't exist, your code raises a `TypeError`. None of these are model failures in the traditional sense — the model did its job and produced plausible text. The failure is that you trusted plausible text to be valid input to a real function. In a one-shot prompt that's an annoyance; in an agentic loop, that single malformed call breaks the Observation step, which corrupts the next Thought, which sends the agent down a path you can't recover from. The loop doesn't crash loudly. It drifts.

The mental model worth holding is this: a tool is not a function. A tool is a contract. The contract has three parts — a name the model can reference, a description that tells the model when and why to use it, and a schema that constrains what the arguments must look like. The Python function behind it is just the implementation detail. Everything the model sees and reasons about is the contract, which means the quality of the contract determines whether the agent makes smart choices or hallucinates. A vague description ("does stuff with users") produces vague usage. An overly clever schema with optional fields and union types produces creative misuse. The contract is your interface to a non-deterministic caller, and it should be designed with the same care you'd give a public API.

Two technologies do most of the work of making this contract enforceable. Pydantic gives you type-safe schemas in Python: you declare what a tool's input looks like with type hints, and Pydantic generates the JSON Schema the model needs, validates the model's output against it after the fact, and gives you typed objects in your function body. It's the translation layer between "Python function" and "thing the model can reason about." The second technology — Structured Outputs, sometimes called Constrained Decoding — goes further. Instead of letting the model generate freely and then validating, it intervenes at generation time, mathematically restricting the model's next-token choices to only those tokens that keep the output conformant to the schema. The model literally cannot emit invalid JSON because the invalid tokens are masked out of the probability distribution. This sounds like a small engineering detail. It is the difference between "my agent works most of the time" and "my agent works."

The progression most teams go through is predictable and painful. First, you parse model output with `json.loads` and a `try/except`. It works in the demo. Then you ship, and you discover that maybe one in fifty calls produces malformed JSON — a stray markdown fence, an unescaped quote, a hallucinated field. You add regex cleanup. You add retries. You add a "please respond in valid JSON" reminder to the system message. The failure rate drops but never to zero, and when it does fail, the agent is now in a state where it doesn't know its last action didn't execute, and it reasons forward as if it had. Adopting Pydantic plus structured outputs collapses this entire category of bugs at once. The model is constrained to produce valid input; your function receives a typed object; validation errors, when they happen, are explicit and can be fed back to the model as an Observation — "your arguments failed validation because X" — letting the agent recover within the loop instead of silently breaking it.

The remaining design work is qualitative. Tool descriptions are prompts in disguise: they are read by the model every turn, and they determine whether it picks the right tool from a list of ten. Too sparse and the model guesses; too verbose and it gets confused or wastes context. Parameter names matter — `user_id` and `customer_identifier` are not interchangeable to a model that's pattern-matching on the surface form. Optional parameters introduce ambiguity about defaults. Returning errors as structured strings that the agent can read and reason about ("not found: no user with id 42") is more useful than raising exceptions that crash the loop. These are small choices, but they accumulate. A well-designed tool surface makes a mediocre model look smart. A badly designed one makes a frontier model look broken.

The takeaway is that schema enforcement is not a polish step you add at the end. It is the foundation that makes the entire Thought-Action-Observation loop reliable enough to iterate on. Until the model's tool calls are guaranteed to parse and execute, you cannot tell whether your agent's failures are reasoning failures, tool design failures, or just JSON failures — and you'll spend your debugging time on the wrong layer. Lock the contract down first. Then the interesting agentic problems become visible.

## Level 2 candidates

**Tool Signature Design** — Covers how to choose tool names, write parameter lists, and craft descriptions that guide the model toward correct usage. Worth deeper exploration because the qualitative choices here — naming, granularity of arguments, return shape — are where most "the model picked the wrong tool" bugs actually originate, and there are concrete patterns and anti-patterns worth cataloguing.

**Pydantic Schemas and Type Safety** — Covers how to use Python type hints and Pydantic models to generate JSON Schemas, validate model output, and surface typed objects inside your tool functions. Worth deeper exploration because Pydantic has nuance (validators, discriminated unions, nested models, field aliases) that maps directly onto agentic patterns and rewards careful study.

**Constrained Decoding and Structured Outputs** — Covers the mechanism by which models are forced to respect a schema at generation time by masking invalid tokens from the probability distribution. Worth deeper exploration because understanding *how* this works — and where it doesn't (streaming, certain providers, complex grammars) — changes how you design schemas and how you reason about failure modes.

**Parameter Validation and Error Handling** — Covers what happens when validation fails despite your best efforts, and how to return errors as Observations the agent can read and recover from. Worth deeper exploration because the difference between "raise an exception" and "return a structured error message" is the difference between an agent that recovers and one that loops or crashes.

**Tool Descriptions and Semantic Clarity** — Covers how to write descriptions that are detailed enough to guide selection but concise enough not to bloat context or confuse the model. Worth deeper exploration because tool descriptions are a hidden form of prompt engineering that gets evaluated every turn, and there's real craft in writing them — including how to disambiguate between similar tools and how to encode usage constraints.
