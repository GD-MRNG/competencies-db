# L1-07 · Feature Stores and Data Pipelines

The notebook lied to you. When you computed "average rating over the last 30 days" inline, three lines above your `model.fit()` call, the number was correct — but only for that moment, in that kernel, with that dataframe in memory. The day you ship the model, something else has to compute that same number, the same way, against fresher data, fast enough to serve a prediction. If those two computations diverge by even a little — a different cutoff time, a different null-handling rule, a different definition of "rating" — your model will quietly make worse decisions in production than it did in your notebook, and you will not know why. This is the problem feature stores exist to solve.

Start with what a feature actually is. A feature is a computed value derived from raw data and attached to an entity you make predictions about. Population of a neighborhood (entity: neighborhood). Days since last purchase (entity: customer). Average rating over the last 30 days (entity: product). The raw data lives in transactional systems, logs, and warehouses. The feature is what your model actually consumes. Everything between those two — the SQL, the joins, the windowing, the aggregation, the null handling — is a pipeline. The question is not whether you have pipelines. You always have pipelines. The question is whether they are written down once and reused, or rewritten three times in three places and slowly drifting apart.

The 2023 rhythm was: pull data into a notebook, transform it, train a model, then — when it came time to deploy — rewrite the transformation logic in whatever language the serving system spoke. This is where bugs live. The training pipeline filtered out test users; the serving pipeline did not. The training pipeline used UTC; the serving pipeline used local time. The training pipeline's "30 days" included today; the serving pipeline's did not. Each of these gaps is a small bug, and each one corresponds to the model performing worse online than offline for reasons no one can immediately diagnose. The industry name for this category of failure is training-serving skew, and it is the single most common reason production models underperform their offline metrics.

A feature store is the architectural answer. It is a system that computes a feature once, from a single definition, and then serves the same value to both training (where you need historical values, point-in-time correct, for every example in your dataset) and inference (where you need the current value, fast, for the entity you are scoring right now). The feature is defined in one place. The computation runs on a schedule or in response to events. The output is stored, versioned, and indexed both by entity and by time. When you train, you ask the store for the value of this feature for this customer as of this timestamp. When you serve, you ask for the value as of now. Same definition, same code, same number.

Once you have this discipline, several things become possible that were painful before. Experimentation gets faster because features are precomputed and shared — you do not rewrite the same "days since last purchase" logic for the fourth model this quarter. Reproducibility becomes real because the feature value attached to a training example is the value that existed at that point in time, not the value that exists now (this is called point-in-time correctness, and it is the thing that prevents data leakage from the future contaminating your training set). Reuse becomes the default — a feature one team builds for churn prediction is available, immediately, to the team building lifetime value forecasts. And monitoring becomes possible at the feature level, not just the model level: you can detect that a feature has gone stale or skewed before any of the models that depend on it start misbehaving.

The pipeline side of this is not glamorous, but it is where the real work happens. You will spend more of your time deciding whether your pipeline is idempotent (can I rerun it without producing different answers?) than tuning your model. You will think harder about incremental computation (how do I update yesterday's snapshot without recomputing the last two years?) than about hyperparameters. You will care about data lineage (when someone changes the definition of "active user" upstream, which features and which models break?) because that question, asked at 9pm during an incident, is the one that decides how long the incident lasts. The shift from ETL to ELT — extract and load the raw data into the warehouse, then transform it there — is part of this same story: keeping transformation logic in a place where it can be inspected, versioned, and rerun, rather than scattered across notebooks and one-off scripts.

The skill this topic builds is the ability to think about features as products with lifecycles, not as cells in a notebook. A feature has a definition, an owner, a freshness guarantee, a schema, a set of downstream consumers, and a versioning history. When you treat features that way, models become reproducible artifacts instead of one-off experiments. When you do not, every deployment is a small act of faith that the numbers in production resemble the numbers you trained on. Feature stores and pipelines are the boring infrastructure that turns data science from a craft practiced once into a system that runs.

## Level 2 candidates

**ETL vs. ELT and idempotency** — Covers why modern pipelines load raw data into the warehouse first and transform it there, and what it means for a transformation to be safe to rerun. Worth a deeper look because idempotency is the property that distinguishes pipelines you trust from pipelines you babysit, and the design choices behind it (deterministic timestamps, partition overwrites, surrogate keys) are not obvious until you've been burned.

**Incremental computation and slowly changing dimensions** — How to update features daily without reprocessing all of history, and how to handle entities whose attributes change while you still need the old values. This earns its own treatment because the naive approach (recompute everything) stops working the moment your data grows, and the correct approach (SCD Type 2, watermarks, late-arriving data handling) has surprising depth.

**Point-in-time correctness and data leakage** — How feature stores reconstruct the value a feature had at a specific historical moment, and why training on "as-of-now" values silently leaks future information into your model. Worth depth because this is the most common subtle bug in real ML systems and the mechanics of avoiding it (event-time joins, asof joins) are non-trivial.

**Data quality monitoring** — How to detect that an upstream source has gone stale, that a column's distribution has shifted, or that null rates have spiked, before any model consuming that data degrades. Deserves a deep dive because more production models fail from data going bad than from algorithms being wrong, and the monitoring patterns (expectations, freshness SLAs, distribution checks) are a discipline of their own.

**Feature versioning and definition drift** — Why "days since last purchase" can mean three different things over a model's lifetime, and how to track definition changes so you can answer which version of which feature trained which model. Worth depth because reproducibility breaks here first, and the tooling answer (feature definitions as code, versioned and reviewed) shapes how teams collaborate.

**Data lineage and governance** — How to answer "where did this number come from?" across a graph of pipelines, tables, and models, and what breaks when an upstream column is renamed or deprecated. Worth its own treatment because lineage is what lets you triage incidents and respond to schema changes without playing detective every time.
