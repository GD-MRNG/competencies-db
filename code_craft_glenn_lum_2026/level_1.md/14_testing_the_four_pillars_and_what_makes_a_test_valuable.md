# L1-14 · Testing: The Four Pillars and What Makes a Test Valuable

Most arguments about testing are arguments about the wrong thing. Coverage percentages, unit-versus-integration ratios, whether to mock the database — these are downstream questions, and you cannot answer any of them well without a prior frame for what makes a single test *worth keeping*. Without that frame, a test suite grows by accretion: every bug spawns a regression test, every code review demands "more tests," and the suite becomes a slow, brittle tax that punishes the team for changing the code it was supposed to protect. The shift that fixes this is small but total. You stop asking whether code is tested and start asking whether each test earns its place.

The cleanest answer the field has produced is Khorikov's four pillars. A valuable test does four things at once: it protects against regressions (it catches real bugs when they're introduced), it resists refactoring (it doesn't break when you change implementation details without changing behaviour), it gives fast feedback (it runs quickly enough that you actually run it), and it's maintainable (you can read it, understand what it asserts, and update it without dread). Every test you write is some combination of these four properties, and the central insight — the one that reorganises everything else — is that they are in tension. You cannot maximise all four. An end-to-end test through the real system has excellent protection and refactoring-resistance but terrible feedback speed. A heavily mocked unit test runs instantly but couples itself to the implementation it was supposed to verify against. Every test is a tradeoff, and the skill is choosing the tradeoff deliberately.

The most under-appreciated of the four is resistance to refactoring, and it is where most test suites silently fail. A test that breaks every time you rename a method, extract a helper, or restructure a class — without any change to what the code actually *does* — is worse than no test at all. It produces false alarms, and false alarms train the team to ignore failures or, worse, to stop refactoring. The whole point of having tests is to make change safe; a test that punishes change has inverted its own purpose. You can tell you have this problem when "the tests broke" is met with a shrug and a mechanical update rather than an investigation. At that point the suite has stopped catching regressions and started enforcing a particular implementation.

This is why the durable rule across the canon is to test behaviour, not implementation. Couple your tests to what the code does as observed from outside — its inputs, its outputs, its visible effects on the world — not to how it does it internally. A test that calls a public method and asserts on the result will survive any refactoring that preserves that result, which is exactly the refactoring you want to encourage. A test that asserts "this private helper was called twice with these arguments" has frozen the implementation in place; the moment you find a better way to compute the same answer, the test fails for no real reason. Behaviour is the contract; implementation is the thing you're trying to keep free to change. Tests should defend the first and ignore the second.

Sitting on top of all this is Cohn's test pyramid, which is best understood not as a rule about ratios but as a tradeoff strategy expressed visually. Many fast, narrow unit tests at the base; fewer, slower integration tests in the middle; very few end-to-end tests at the top. The pyramid shape is a direct consequence of the four pillars: as you move up, protection and realism increase but feedback speed and maintainability decay quickly, so you want most of your tests where the cost is lowest and reserve the expensive ones for the integration points that genuinely require them. The inverted pyramid — heavy on slow end-to-end tests, light on units — is the suite that takes forty minutes to run, fails intermittently, and that nobody trusts. The hourglass and the ice-cream cone are both ways of forgetting that every test you keep is paid for on every run.

Once you hold the pillars, the contested questions in testing become tractable rather than tribal. "Should I aim for 100% coverage?" — no, because coverage measures only one pillar (a weak proxy for protection) and ignores the others entirely; a 100%-covered codebase full of implementation-coupled tests is a codebase you cannot refactor. "Should I mock this collaborator?" — only if mocking improves the tradeoff; if it trades fast feedback for a permanent loss of refactoring-resistance, you've made the suite worse. "Is this test worth writing?" — apply the four pillars and see whether it scores well enough on any of them to justify its maintenance cost. The questions don't change; you just have a way to answer them.

The deepest payoff of this topic is what it unlocks elsewhere on the map. Refactoring is only safe when you have tests you can trust; tests you can trust are tests built on behaviour, not implementation; tests built on behaviour are the ones the four pillars produce. A team that gets this right has a refactoring safety net and can keep paying down complexity for years. A team that gets it wrong has a suite that ossifies the code it was supposed to liberate. The difference between those two teams is rarely how much they test. It's whether they can tell, for any given test, what it's actually worth.

## Level 2 candidates

**The Four Pillars and Their Tradeoff** — Walks through protection, refactoring-resistance, fast feedback, and maintainability as competing properties no test maxes simultaneously. Worth depth because the *tension* is the whole insight, and seeing how specific test styles score on each pillar is what converts the frame from a slogan into a working judgment.

**Resistance to Refactoring: The Forgotten Pillar** — Drills into why a test that breaks on implementation changes is net-negative, and how to recognise the failure mode in an existing suite. Worth depth because this is the pillar most often missing from inherited testing intuitions, and naming it clearly is what diagnoses the "our tests are too brittle" complaint.

**Test Behaviour, Not Implementation** — Examines what "behaviour" actually means at different levels (unit, module, system) and how to write assertions that survive restructuring. Worth depth because the principle is easy to state and hard to apply — most teams who agree with it in the abstract still write implementation-coupled tests in practice.

**The Test Pyramid and Where Coverage Pays** — Unpacks Cohn's pyramid as a consequence of the four-pillars tradeoff rather than a rule about ratios, and addresses why coverage targets are the wrong goal. Worth depth because the alternatives (inverted pyramid, ice-cream cone) are common enough to deserve named diagnosis, and because the "what should I aim for instead of 100%?" question needs a concrete answer.

**Mocks, Stubs, and Over-Mocking** — Covers when test doubles genuinely help isolation versus when they freeze the design and couple tests to call structure. Worth depth because mocking is where the four pillars most visibly conflict, and because the heuristic that emerges (mock at process boundaries, not internal collaborators) is the practical rule that prevents most brittle-test pathologies.
