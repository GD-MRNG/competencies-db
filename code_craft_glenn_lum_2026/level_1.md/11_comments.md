# L1-11 · Comments

There is a generation of programmers who were taught, sincerely and often as their first principle of style, that a comment is a confession of failure. The reasoning sounds airtight: if the code were good enough, it would explain itself, so a comment is a patch over a deficiency you should have fixed in the code. This is one of the most thoroughly absorbed lessons in modern software craft, and it is half right in a way that has done a great deal of quiet damage. The half it gets wrong is that some of the most important things a future reader needs to know cannot, even in principle, be expressed in code — and a discipline that calls those comments failures simply trains people to throw the information away.

The schism between the two camps is real and worth naming clearly. On one side sits a strand of agile folk wisdom and parts of Clean Code: comments rot, comments lie, comments are an excuse for unclear naming, and the right move is almost always to refactor until the code reads as prose. On the other side sits Ousterhout, taking the opposite strong position — that comments are not a failure but a category of information the code itself is incapable of carrying, and that a codebase without them is not self-documenting but self-deceiving. He goes further still, arguing that you should write the interface comment first, before the implementation, because the act of explaining what a method is for in plain language exposes muddled designs while they are still cheap to fix.

The reason this argument has run so long and so hot is that both sides are pointing at real comments and drawing opposite conclusions because they are not, in fact, talking about the same thing. The dissolution of the fight is to notice that "comment" is a category that bundles two very different artifacts together. A comment that restates what the code plainly does — a loop counter incrementing, a variable being assigned, a function whose name already says what it returns — is genuinely noise. It adds reading cost, it drifts out of sync, and the self-documenting-code camp is correct that the right fix is to delete it and, if needed, improve the name. But a comment that records why a decision was made, which alternative was rejected and on what grounds, what invariant the caller must maintain, or what non-obvious constraint forced this exact shape — that comment is carrying information that exists nowhere else in the system and will be lost the moment it isn't written down.

That distinction is the working test you should carry into every code review. Ask of any comment: is this telling me something I could have read off the code in the next two seconds, or is it telling me something the code physically cannot say? The first kind you delete without ceremony. The second kind is load-bearing — the kind of thing that, when missing, causes a future maintainer to "fix" something that wasn't broken, or to spend a day rediscovering a constraint the original author already paid to learn. Code can show you what; it struggles to show you why; it cannot show you what was considered and rejected, or what would silently break if you changed this line.

Once you accept that the why-comment is irreplaceable, Ousterhout's stronger move — write the comments first — stops sounding eccentric. The interface comment for a function is, before anything else, a statement of what the function is for and what its contract is. If you cannot write that paragraph cleanly, the interface is not clean, and you have been handed that diagnosis at the cheapest possible moment, before any implementation has been built around the confusion. Comments-first is a design technique disguised as a documentation habit, and the habit pays for itself the first time it makes you delete a method you were about to write because you couldn't say what it did without three caveats.

The practical objection to all of this is that comments rot — they drift from the code because nothing enforces their truth, and a wrong comment is worse than no comment at all. This is true, but it is a guide to which comments to write rather than a reason not to write any. Comments rot in proportion to how tightly they are coupled to the volatile detail of the implementation; comments about the why, the rationale, the constraint imposed from outside, the alternative rejected for a documented reason — those things change far more slowly than the code that implements them, because they are facts about the problem, not facts about this week's solution. The comment that says "increments i" rots the moment the loop becomes a map; the comment that says "we sort here because the downstream API requires ascending order and silently truncates otherwise" is still true ten refactors later.

There is also a clean split worth keeping inside your head between interface comments and implementation comments, because they serve different readers and have opposite obligations. An interface comment is for the caller and should hide internals — it tells you what to expect and what you owe, not how the method achieves it. An implementation comment is for the maintainer working inside the function and is allowed, even obliged, to talk about the messy middle: why this branch exists, why this order matters, why the obvious approach was tried and abandoned. Confusing the two is how you end up with interface documentation that leaks the implementation it was meant to hide, forcing every caller to learn details that should have been the module's secret.

What this topic ultimately unlocks is a posture shift: you stop treating comments as apologies for code that wasn't clean enough and start treating them as design artifacts in their own right — some load-bearing, some deletable, and your job is to know which is which. The rule is not "comment everything" or "comment nothing." The rule is that the why, the rationale, and the non-obvious constraint live in comments because they have nowhere else to live, and the what lives in the code because that is the only place it belongs. A reviewer who deletes the first kind is doing the codebase harm; a reviewer who tolerates the second kind is doing it harm of a different sort. Knowing the difference is the whole skill.

## Level 2 candidates

**The Debate: Self-Documenting Code vs Comments as Essential** — Lays out the two camps in their strongest forms — Clean Code's "good code needs no comments" against Ousterhout's "comments capture what code cannot" — and shows that they are defensible about different comments rather than genuinely opposed. Worth a deep dive because seeing both arguments in their best version is what stops a reader from defaulting to whichever they read first.

**Comments Should Say What the Code Cannot** — The operational rule for telling a useless comment from an irreplaceable one: the why, the rationale, the rejected alternative, the non-obvious constraint, all live nowhere in the code itself. Worth deeper treatment because the rule is simple to state and surprisingly hard to apply consistently in review without worked examples.

**Write the Comments First** — Ousterhout's claim that drafting the interface comment before the implementation is a design technique, not just a documentation one, because it surfaces confused designs while they are still cheap to fix. Worth depth because the practice is unfamiliar to most developers and only convinces in the doing — the post would walk through what a comment-first session actually looks like.

**Why Comments Rot, and What to Do About It** — The mechanism by which comments drift from code (nothing enforces their truth) and the design response (comment the stable why, not the volatile how). Worth depth because rot is the strongest argument the no-comments camp has, and answering it directly is what makes the pro-comment position credible in practice.

**Documenting the Interface vs the Implementation** — Two different comment types serving two different readers, with opposite obligations about what to reveal and what to hide. Worth a separate post because the leak-the-implementation failure mode is common and most style guides do not name it crisply enough to prevent it.
