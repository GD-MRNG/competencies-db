# Programming Fundamentals

Most practitioners with a few years of experience can write code that works without ever fully understanding what the machine is doing in response. This is fine until it isn't. The bug that takes a week to track down, the performance cliff that appears under load, the off-by-one that survives review — these almost always live not in the logic of the code, but in the gap between what the programmer thought a construct meant and what it actually translates to in memory and execution. Programming fundamentals is the layer that closes that gap.

The premise is that a programming language is a contract with two parties. One party is the human reader, who needs to understand intent. The other is the machine, which needs precise instructions about what to allocate, what to read, what to branch on, and what to return. Most teaching emphasises the first party — readability, idioms, style. Most production failures come from misunderstanding the second. The fundamentals are the small set of concepts that, once internalised, let you predict what the machine will do from a glance at the code. Without them, you are debugging by intuition. With them, you are debugging by deduction.

The first concept is the variable, which is more subtle than it appears. A variable is a name bound to a piece of memory holding a value. Whether that name refers to the value itself or to a reference pointing somewhere else is the distinction that separates languages, and within a single language often separates types. When you assign one variable to another, do you copy the value or share the reference? When you pass an argument to a function, does the function receive its own copy or a handle to yours? These are not trivia questions. The answer determines whether your changes propagate, whether your data structures alias each other, and whether mutation in one place silently breaks invariants somewhere else. Layered onto this is the question of where the memory itself lives — on the stack, where it is automatically reclaimed when a function returns, or on the heap, where it persists until something explicitly or implicitly frees it. Most subtle bugs in any language with mutable state live in this layer.

Control flow is the next concept, and the one most practitioners think they understand best. The mental model worth having is that conditionals and loops are implementations of logical propositions. An if-statement is a proposition the program is checking before deciding what to do; a while-loop is a proposition the program will keep checking until it becomes false. Framed this way, control flow stops being a syntactic feature and becomes a tool for encoding logical structure. The practical consequence is determinism: a program whose control flow is well-defined will behave the same way given the same inputs, and a program whose control flow depends on undefined orderings — concurrency, uninitialised memory, iteration over unordered collections — will not. Most "flaky" behaviour in production traces back to a control flow assumption that holds almost always but not quite always.

Functions are the third concept, and they are the first real act of abstraction the programmer performs. A function is not just a way to avoid repeating yourself. It is a contract: given these inputs, I will produce this output, and the rest of the program does not need to know how. This contract is what allows large programs to be composed at all. Without it, every change ripples through every line that depends on it. With it, you can change the inside of a function freely as long as the contract holds. Tied to functions is the concept of scope — what names are visible where, and how long they live. Scope is what makes the contract enforceable: a function's internals are private precisely because the names it uses do not leak into the caller. When scope rules are unclear or violated (global state, leaky closures, shared mutable references), the contract breaks down, and reasoning about the program becomes vastly harder.

Recursion is the fourth concept, and it is the one most practitioners use uneasily even when they use it correctly. A recursive function is one that calls itself with a smaller version of its input until it hits a base case. The reason it feels unintuitive is that imperative thinking is sequential — do this, then this, then this — while recursive thinking is structural. You define what the answer is for the smallest case, and you define how the answer for a larger case relates to the answer for a smaller one. This is mathematical induction, expressed as code. It also happens to be the natural way to express any problem with a recursive structure: trees, grammars, nested data, divide-and-conquer algorithms. Once you see recursion as induction, the apparent magic disappears, and a whole class of problems becomes obviously easier to solve recursively than iteratively.

Taken together, these four concepts form a closed loop. Variables and memory describe what the program holds. Control flow describes how it decides what to do next. Functions describe how complexity is partitioned. Recursion describes how a function can be defined in terms of itself, which turns out to be the foundation of how compilers, parsers, and many algorithms are naturally expressed. Every higher-level construct in every language — classes, modules, async, generics, monads — is built on top of this loop. You cannot reason about the higher constructs if you do not have the lower ones solid.

The skill this topic builds is the ability to read a piece of code and predict, before running it, what will happen in memory and in execution. Not the output — the mechanics. That predictive ability is what separates a practitioner who debugs by reasoning from one who debugs by guessing.

## Level 2 candidates

**Variables, types, and memory** — Covers what a variable actually is at the machine level: the distinction between values and references, stack versus heap allocation, and how type declarations constrain the layout and behaviour of memory. Worth a deep dive because this is the layer where most subtle bugs live, and the mental model transfers directly into understanding pointers, ownership models, garbage collection, and most debates about language design.

**Control flow** — Covers conditionals, loops, and the less obvious mechanics of how a program decides what to do next, including how non-determinism enters through concurrency, exceptions, and iteration order. Worth a deep dive because the framing of control flow as encoded logical propositions reframes a lot of imperative code and makes the failure modes of "flaky" behaviour comprehensible.

**Functions and scope** — Covers function definition, parameter passing, return values, closures, and the rules that determine where names are visible and for how long. Worth a deep dive because scope is the mechanism that makes abstraction enforceable, and most bugs in larger codebases come from scope assumptions that do not hold (shared mutable state, leaky closures, accidental globals).

**Recursion** — Covers recursive function definition, base cases, the call stack, and the relationship between recursion and mathematical induction. Worth a deep dive because recursion is the natural expression of any problem with self-similar structure, and seeing it as induction-in-code unlocks tree algorithms, parsers, and divide-and-conquer methods that look opaque when forced into iterative form.
