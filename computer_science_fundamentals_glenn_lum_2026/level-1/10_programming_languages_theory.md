# L1-10 · Programming Languages Theory

You have written code in five, maybe ten languages. You have opinions about which ones you like. What you probably do not have is a clear answer to the question: why do these languages exist at all? Not "why was Python invented" in the historical sense, but why is there a Python-shaped hole in the space of possible languages, and a Rust-shaped hole, and a Haskell-shaped hole? Languages are not a matter of taste or branding. Each one is a position on a small number of deep questions about what programs are and what kinds of mistakes the language is willing to let you make.

The first move is to stop thinking of a language as a syntax. Syntax is the surface — the curly braces, the indentation rules, the keyword choices — and it is the least interesting part. Underneath the syntax is a grammar (a formal specification of what strings of characters even count as a program), and underneath the grammar is a model of computation: a theory of what a program is doing when it runs. Two languages with nearly identical syntax can have radically different models, and two languages that look nothing alike on the page can be doing essentially the same thing. The mental model you want is that a language is a contract between you and the machine about which questions are answered when, and which categories of error are possible at all.

The "when" part of that contract is what type systems formalise. A type system is a mechanism for proving things about your program before it runs — that this variable is a number, that this function will not be called with a string, that this list is never empty. Static type systems do this proof at compile time and refuse to produce a running program if the proof fails. Dynamic type systems defer the same checks to runtime, where they manifest as exceptions. Strong versus weak type systems is a separate axis: how aggressively the language coerces values across type boundaries. The tradeoff is not "safety versus convenience" — it is "which errors do you want to find in five seconds versus five months." Most serious language design debates reduce to disagreements about where on this axis a working programmer should sit.

Then there is the question of how source code becomes execution. Compilers translate your code, ahead of time, into something the machine or a virtual machine can run directly. Interpreters walk the source (or a parsed representation of it) and execute it as they go. This distinction sounds clean but blurs immediately in practice — most modern languages do some compilation, even if it is just to bytecode, and most "compiled" languages have interpreted modes for development. The compilation pipeline is roughly the same regardless: lex the source into tokens, parse the tokens into an abstract syntax tree, analyse the tree (type-check it, resolve names, check scopes), optimise the resulting representation, and emit something runnable. The performance of a language is not really decided by whether it is "compiled" or "interpreted." It is decided by how much the language permits a compiler to know about your program in advance, which loops back to the type system.

Memory is the next deep axis. Every language has to answer: when you create a value, where does it live, and who is responsible for cleaning it up? C says: the programmer is responsible, and if you get it wrong, the program corrupts itself. Java says: the runtime is responsible, and a garbage collector will pause your program periodically to figure out what is no longer reachable. Rust says: the compiler is responsible, and it will refuse to compile your program unless ownership of every value is statically clear. These are not three implementations of the same idea — they are three different theories of what safety means and who pays for it. The choice ripples through the entire feel of the language: what concurrency primitives are sane, how cheap function calls are, how predictable performance is.

The last big axis is paradigm. Imperative languages describe computation as a sequence of state changes: do this, then this, then this. Functional languages describe computation as the evaluation of expressions: this value is the result of applying these functions to these inputs, and side effects are either forbidden or carefully quarantined. The difference is not cosmetic. An imperative loop and a functional fold are not just two ways to write the same thing; they are two different claims about what the program is. Most modern languages now borrow from both — JavaScript has map and filter, Java has lambdas, Python has comprehensions — but the borrowing makes more sense once you have seen each paradigm in something close to its pure form. The borrowing is not a synthesis; it is a compromise, and the seams show.

What you get from understanding all this is not the ability to design a language. You will probably never design one. What you get is the ability to read a language's design, see what it is optimising for, and predict what kinds of bugs you will hit and what kinds you will not. When a language frustrates you, you will be able to say specifically what tradeoff you are paying for, rather than generally complaining that it is verbose or magical or slow. And when you are choosing a tool — for a service, for a script, for a system that needs to last a decade — you will be choosing on the basis of what the language actually guarantees, not on the basis of what its users say in conference talks.

## Level 2 candidates

**Syntax and grammars** — How languages are formally specified, including context-free grammars and BNF notation, and how the grammar determines what programs are even expressible. Worth going deeper because grammar choices encode design decisions about ambiguity, parsing complexity, and which categories of error become syntactically impossible — and because reading a language spec requires fluency here.

**Parsing** — The mechanics of turning a stream of characters into tokens and then into an abstract syntax tree the rest of the toolchain operates on. Worth going deeper because every language tool you use — linters, formatters, IDEs, compilers — is built on top of a parser, and the techniques (recursive descent, parser generators, PEG) recur in any structured-text problem.

**Type systems** — The full landscape of static versus dynamic, strong versus weak, structural versus nominal, and what each combination buys and costs. Worth going deeper because type systems are where most of the recent action in language design has happened, and because the conceptual machinery (types as proofs, inference, parametric polymorphism) shows up in domains far beyond programming languages.

**Compilers vs interpreters** — The full compilation pipeline (parse, analyse, optimise, generate) and the spectrum between pure interpretation, JIT compilation, and ahead-of-time compilation. Worth going deeper because performance characteristics, debugging behaviour, and deployment models all flow from where on this spectrum a language sits.

**Memory models** — Stack versus heap allocation, garbage collection strategies, manual management, and ownership-based systems like Rust's. Worth going deeper because the memory model determines what concurrency looks like, what real-time guarantees are possible, and where a language can and cannot be used — and because the tradeoffs are not obvious until you have seen the failure modes of each.

**Functional vs imperative paradigms** — The deep distinction between describing computation as state change versus as expression evaluation, and what referential transparency, immutability, and higher-order functions actually buy you. Worth going deeper because most modern languages are hybrids, and the borrowing only makes sense once you have seen each paradigm in a relatively pure form.
