# L1-12 · Multi-Agent Coordination

The instinct, once you have one coding agent working reliably, is to run several. If one agent can ship a feature in an afternoon, surely four agents can ship four features — or, more ambitiously, decompose one feature into parallel subtasks and finish in a quarter of the time. This intuition is wrong often enough to be dangerous. Multi-agent systems are not a free multiplier on a working single-agent workflow; they are a different system with different failure modes, and most of the time the bottleneck they claim to solve isn't the one you actually have.

Start with what coordination costs. The moment you have more than one agent acting on a shared artifact — a codebase, a database, a deployment target — you have introduced a distributed system, with all the attendant problems: synchronisation, conflicting writes, stale views of state, and emergent behaviour that no single agent's logs can explain. None of these problems exist in a single-agent loop. You are paying for them in tokens (the orchestration prompts, the inter-agent messages, the redundant context each agent needs) and in legibility (you now have N reasoning traces to reconcile instead of one). The question is whether what you get back exceeds what you spend, and the honest answer is: usually only when the underlying task is genuinely parallelisable and the single agent was the bottleneck.

That second condition matters more than people realise. Capability saturation is the phenomenon where, above a certain single-agent capability threshold on a given task, adding more agents stops helping and starts hurting. If one well-configured agent can already complete the task reliably, decomposing it across multiple agents introduces coordination overhead with nothing to offset it. The orchestrator has to split the work, the subagents have to be briefed, their outputs have to be merged, and each handoff is a place where context is lost or contradicted. You can end up with a system that is slower, more expensive, and less reliable than the single agent you started with — while feeling more sophisticated, which is the worst combination.

The dominant pattern in practice is orchestrator-subagent: one agent decomposes a task and delegates pieces to others, then assembles the results. This works when the decomposition is clean — independent subtasks with well-defined interfaces and minimal shared state. It breaks down when the subtasks turn out to be coupled in ways the orchestrator didn't anticipate, which is most non-trivial software work. Two subagents editing different files in the same module can each produce locally coherent changes that are globally incompatible: the function signature one of them changed is the function the other one called. Neither agent's tests fail in isolation. The merge produces a broken system that no individual reasoning trace explains, because the failure lives in the gap between agents.

This is the shared-context problem, and it is the multi-agent failure mode you should expect rather than be surprised by. Each agent has its own context window, its own view of the codebase at the moment it started, and its own sense of what "done" looks like for its slice. Coordination protocols can mitigate this — locking, sequential phases, explicit interfaces — but every mitigation is a constraint that reduces the parallelism you ran multiple agents to get. The limit case of a perfectly-coordinated multi-agent system is a single agent with extra steps.

Trust complexity compounds the problem. Everything from the trust boundaries chapter — prompt injection, supply chain risk, the inability to distinguish data from instruction — applies to each agent in the system, and now also applies to the messages agents send each other. A successful injection in one agent can propagate through the inter-agent channel into others, because the receiving agent treats the sender's output as trusted instruction. The blast radius of any single compromise is no longer one agent's permission scope; it is the union of all the agents that trust it. This is not theoretical. It is the predictable consequence of treating inter-agent messages as a privileged channel without enforcing the same skepticism you would apply to any other input.

The practical posture is therefore conservative. Before reaching for multi-agent coordination, identify the actual bottleneck. If the bottleneck is your single agent's reliability on the task, fix that first — better scaffold, better feedback environment, better specification — because none of those problems get easier when you add more agents. If the bottleneck is genuinely throughput on independent, well-bounded tasks (running the same kind of refactor across many isolated files, say, or generating boilerplate for a set of unrelated endpoints), then parallel agents pay off, and the coordination problem is small because there's nothing to coordinate. The cases in between — coupled work that you wish were parallelisable — are where most of the disappointment lives.

The skill this topic builds is recognising that "more agents" is rarely the right answer to "this isn't working." The right answer is almost always upstream: the loop, the scaffold, the feedback environment, the specification. Multi-agent coordination is a tool for a specific shape of problem, not a general scaling strategy. Reach for it when you can name the bottleneck it solves and articulate why a single agent cannot solve it. If you cannot, you are buying complexity you have not yet earned.

Worth deeper treatment because the pattern is widely deployed but its failure modes — coupled subtasks, interface drift, orchestrator hallucination of subagent capabilities — are rarely discussed alongside the marketing.

Worth a deep dive because the threshold is task-specific and the standard advice ("just add more agents") gives you no way to detect when you've overshot.

**Shared-state and merge conflict patterns** — Covers the specific failure modes that emerge when multiple agents act on a shared codebase: locally coherent but globally incompatible edits, stale-view conflicts, and the silent class of bugs where no individual agent's tests fail but the merged system is broken. Worth deeper treatment because these failures are diagnostically hard — the reasoning traces of each agent look fine in isolation — and the mitigations (locking, phasing, explicit interfaces) trade off directly against the parallelism you wanted.

**Cross-agent trust propagation** — Covers how prompt injection, context poisoning, and other compromises propagate through inter-agent message channels, and why the receiving agent's treatment of sender output as trusted instruction creates an amplified blast radius. Worth a deep dive because it sits at the intersection of L1-06 and L1-12 and the standard threat model for single-agent systems systematically underestimates it.

**Coordination protocol design** — Covers the explicit mechanisms (locks, phases, interfaces, message schemas) that let multi-agent systems avoid the worst shared-state failures, and the cost each mechanism imposes on the parallelism it protects. Worth deeper treatment because the design space is real but underexplored, and most teams either skip protocols entirely (and discover the failures in production) or over-engineer them (and lose the speedup that motivated multi-agent in the first place).
