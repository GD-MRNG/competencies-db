# L1-01 · The Perception–Planning–Execution Loop

The thing that makes a coding agent an agent — rather than a very good autocomplete — is that it runs in a loop. A code generator takes a prompt and returns a string. An agent takes a goal, looks at the world, decides on an action, performs it, looks at the world again, and repeats until it believes it is done. That distinction sounds small. It is the entire ballgame. Almost every surprising behaviour you will encounter from a coding agent — the confident wrong answer, the infinite retry, the silent drift away from your intent — is a property of the loop, not of the model running inside it.

The loop has three phases, and it is worth naming them precisely because each one fails in its own characteristic way. Perception is the agent ingesting state: reading files, listing directories, running a command and capturing output, receiving the result of a previous tool call. Planning is the agent reasoning about what to do next given what it just perceived — the step where the model produces tokens that commit to an action. Execution is the action itself: an edit, a shell command, a tool invocation, a network call. Then perception happens again, on the world the execution just changed, and the cycle continues. This pattern was formalised for language models as ReAct (Yao et al., 2022), but the lineage runs back through robotics and control theory; the loop is older than the LLM by decades, and the LLM is just an unusually flexible planner sitting inside it.

Once you see the loop, you start seeing the seams between phases, and the seams are where things break. A perception failure means the agent's model of the environment has diverged from reality — it thinks a file contains something it no longer contains, it missed an error in tool output it didn't read carefully, it never looked at the test that would have told it the truth. A planning failure means the agent reasoned correctly about a wrong picture, or reasoned incorrectly about a right one. An execution failure means the action the agent intended is not the action that actually happened — the tool schema was ambiguous, the shell command was malformed, the edit landed in the wrong place. These failures look different in the log but compound the same way: a wrong perception feeds a wrong plan, a wrong plan produces an action that further distorts the environment, and the next perception is now of a world the agent itself has degraded. The loop does not self-correct unless the environment provides a signal sharp enough to puncture the drift.

This is why "the model got it wrong" is almost never a useful diagnosis. The model is one component in a system whose behaviour is determined by the loop's structure: what state the agent can perceive, how that state is encoded between iterations, what actions are available, what feedback those actions return. Swap the model for a smarter one and you accelerate the loop without changing where it fails. If the agent cannot see test output, a better planner will produce more confident code that still doesn't pass. If the tool interface is ambiguous, a better model will hallucinate calls more fluently. The loop is the substrate; the model is a tenant.

Two structural properties of the loop deserve special attention because they show up everywhere downstream. The first is state representation — how the agent encodes what it knows from one iteration to the next. Context windows are finite, so the agent is constantly making implicit decisions about what to keep and what to drop, and the way it represents accumulated knowledge determines whether it can reason across a long task at all. An agent that can hold a coherent picture of a fifty-step refactor is doing something categorically different from one that re-derives its understanding every iteration, even if both use the same model. The second is termination — how the loop decides it's finished. A miscalibrated termination condition produces both of the most expensive failure modes you'll see: the agent that stops too early with a confidently wrong "done," and the agent that cannot stop at all and burns tokens spiralling on a problem it has no traction on. Termination is not a UX detail; it is the mechanism by which an unbounded loop becomes a bounded task.

There is also a real choice about whether to use a loop at all. ReAct-style open-ended loops are flexible — the agent can respond to whatever it perceives — but flexibility is exactly what makes them unpredictable and expensive. Fixed pipelines (perceive these specific things, plan in this specific way, execute this specific sequence) give up adaptability for reliability and cost predictability. Many production "agentic" systems are actually pipelines with a small amount of looping at the edges, because pure ReAct is hard to keep on the rails. Knowing which architecture you are looking at, and where the design has traded flexibility for control, is the first question to ask of any agent product.

The practical skill this builds is locating failures in the loop rather than in the model. When an agent does something wrong, the useful question is not "was the model good enough" but "which phase failed, and why didn't the next iteration catch it." Sometimes the answer is that the agent never perceived the thing it needed to. Sometimes the plan was right but the action was malformed. Sometimes the loop terminated before the signal that would have corrected it arrived. Each of these has a different fix, and none of them is a better model. Once you reason in loop phases, the rest of the agent stack — scaffolds, feedback environments, context strategies, autonomy settings — stops looking like a grab-bag of features and starts looking like what it actually is: a set of design decisions about how to make a particular loop converge on a particular kind of task.

## Level 2 candidates

**Observation grounding** — Covers how the agent's internal model of the environment drifts from the actual state of the world, and why that drift compounds across iterations rather than self-correcting. Worth a Level 2 because the mechanics of grounding failure (what the agent saw, what it inferred, what it assumed) are the single most common root cause of agent misbehaviour, and recognising the signature in a trace is a learned skill.

**ReAct vs. fixed pipelines** — Covers the architectural choice between open-ended loops and structured sequences, and what each gives up. Worth a Level 2 because most real systems sit somewhere on this spectrum and the tradeoffs (flexibility, cost, reliability, debuggability) determine which workloads each architecture can actually carry.

**Loop termination conditions** — Covers how an agent decides it is finished and why miscalibration produces both premature "done" claims and infinite retry spirals. Worth a Level 2 because termination logic is rarely documented, varies sharply across tools, and is the lever that turns an unbounded process into a bounded one.

**State representation between iterations** — Covers how the agent encodes what it knows from one loop step to the next, and why that encoding sets a hard ceiling on how long a task it can actually handle. Worth a Level 2 because it connects the loop directly to context-window economics and is where most "the agent forgot" failures actually originate.

**Phase-transition failure modes** — Covers the specific signatures of perception, planning, and execution failures, and how they compound when the loop continues without correction. Worth a Level 2 because diagnosing agent failures by phase is a transferable skill that survives any specific tool or model, and most teams currently lack a vocabulary for it.
