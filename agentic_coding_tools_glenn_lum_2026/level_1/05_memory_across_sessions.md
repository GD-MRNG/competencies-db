# L1-05 · Memory Across Sessions

Every conversation with an agent starts from zero. Whatever it figured out yesterday — the quirk in your build system, the reason that one test is flaky, the convention you corrected it on three times — is gone unless you did something deliberate to keep it. The context window is a working surface, not a ledger. This is easy to forget because the agent feels continuous within a session: it remembers what you told it ten minutes ago, picks up references, learns your preferences. None of that survives the close of the window. The next session is a stranger wearing the same name.

The first instinct is to treat this as a deficiency to be patched — bigger context, longer conversations, persistent threads. That framing misses what is actually going on. Statelessness is the default condition of the underlying model; persistence is something you engineer on top. Anything the agent "remembers" across sessions exists because somebody — you, the scaffold, or the agent itself — wrote it somewhere durable and arranged for it to be loaded back in. Memory across sessions is not a feature the model has or lacks. It is an externalisation problem, and the choices you make about what to externalise and in what form quietly shape the agent's behaviour more than any individual prompt you write.

This reframes memory as a design surface rather than a limitation. You are deciding, implicitly or explicitly, what the next session of this agent will know about the project. Skip the decision and you get whatever default the tool ships with — usually some autogenerated notes file the agent updates when it feels like it. Make the decision deliberately and you start shaping a small, curated specification of how this codebase wants to be worked on, which the agent reads in at the top of every session.

The fundamental tradeoff in what you persist is between episodic and semantic memory. Episodic memory is the record of what happened: this is the conversation, these are the files that were touched, this is the sequence of tool calls. It is recoverable in detail and high-fidelity, but it is expensive to store and even more expensive to load — replaying yesterday's session into today's context burns tokens on transcript that has mostly nothing to do with the current task. Semantic memory is the distilled version: this is what we learned, this is the convention we settled on, this is the gotcha to remember. It is cheap to load and cheap to act on, but the distillation is lossy, and when something goes wrong you often cannot reconstruct why the agent believes what it believes. Most useful memory systems sit somewhere on the spectrum between them, and most failures come from being unclear about which mode you are actually in.

The other axis worth being explicit about is who is writing the memory. Some of it you author — the instruction file pinned to the repo, the architecture notes, the explicit "do not do X" rules. Some of it the agent writes autonomously — scratchpads, summary files, "lessons learned" notes that some scaffolds generate at the end of a session and re-read at the start of the next. The autonomous artifacts are the ones to watch. They feel like background infrastructure, but they are inputs to every future session, which means they have the authority of a system prompt without the visibility of one. An agent that has quietly recorded an incorrect assumption in its own memory file will keep acting on it, plausibly and confidently, until you read the file and notice. Audit them the way you would audit a configuration file checked into the repo, because functionally that is what they are.

The failure mode this all guards against is cross-session inconsistency, and it is sneakier than it sounds. In a long-running project worked on by an agent across dozens of sessions, decisions made early — the data model, the error handling convention, the way modules talk to each other — silently constrain what is reasonable later. With no memory, the agent in session forty has no idea what was decided in session three and will cheerfully propose a design that contradicts it. The contradiction is locally coherent: the new code makes sense in isolation. It is only broken in relation to the decisions it cannot see. This is the form of technical debt agents accumulate fastest, and it is invisible at review time because each individual diff looks fine.

The practical posture, then, is to treat memory as a thing you curate, not a thing the tool gives you. Decide explicitly what should survive a session: the conventions, the architectural choices, the recurring corrections you find yourself making. Write them somewhere the agent will actually load — the instruction file, the project README, a dedicated notes document referenced from the top of every session. Audit what the agent writes about itself and your project. Treat the persistence layer as part of the codebase, with the same review standards. The agents you find easy to work with over months are not the ones with the cleverest models; they are the ones whose memory you have actually designed.

**Episodic vs. semantic memory** — Covers the tradeoff between persisting raw session history (recoverable, expensive, noisy) and persisting distilled lessons (cheap, lossy, opaque) along with hybrid strategies that attempt to capture both. Worth deeper treatment because the choice of representation determines what kinds of failures are recoverable later, and because most production memory systems make this choice implicitly and inherit its consequences without realising it.

**Auto-generated memory artifacts** — Covers the files agents write to disk between sessions on their own initiative — scratchpads, summaries, lessons-learned notes — and the mechanisms by which they get re-loaded. Worth deeper treatment because these artifacts function as untyped configuration with system-prompt-level authority, and the practices around auditing, versioning, and correcting them are not obvious from the Level 1 framing.

**Cross-session consistency failure** — Covers the specific failure mode where decisions made in early sessions are silently contradicted by later ones because the relevant memory was absent, incomplete, or wrong. Worth deeper treatment because the failure is invisible at diff-review time and only manifests as architectural incoherence over weeks, which means catching it requires habits and tooling that a single Level 1 post cannot build.

**Memory and the instruction-file surface** — Covers the relationship between session memory and the project-level instruction files (CLAUDE.md, AGENTS.md, .cursorrules) introduced in L1-04, including when learned knowledge should be promoted from autogenerated notes into authored instructions. Worth deeper treatment because the boundary between these two surfaces is where most teams either accumulate stale autogenerated cruft or fail to capture genuinely useful learning, and getting it right is a workflow design problem rather than a tooling one.
