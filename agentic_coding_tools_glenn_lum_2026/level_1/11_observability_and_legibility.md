# L1-11 · Observability and Legibility

You can recover from an agent making a bad decision. You cannot recover from an agent making a decision you didn't see. That asymmetry is the entire reason this topic exists, and it is why observability stops being a debugging convenience the moment autonomy crosses any meaningful threshold and becomes the load-bearing control on whether you actually own the code in your repository. The shift is subtle: when a colleague writes code, you trust the artifact because you trust the process that produced it; when an agent writes code, you have no such process to trust, and the only substitute is a record clear enough that you can reconstruct what happened after the fact.

The mental model worth holding is this. Agency and legibility are coupled variables, and the gap between them is liability. A line-by-line autocomplete produces almost no legibility debt because every action is visible at the moment it happens. A multi-hour autonomous session produces enormous legibility debt unless something is actively recording what the agent saw, decided, and did. Most production incidents with coding agents are not "the agent did something wrong" — they are "the agent did something, and nobody can now reconstruct why." The fix is not better models or stricter prompts; it is making the agent's work legible enough that a human can audit it on a timescale that matches the speed at which the agent produced it.

Legibility shows up at four layers, and each one answers a different question. The diff answers what changed. The tool call log answers what the agent touched in the world — which files it read, which commands it ran, which network calls it made, which MCP servers it consulted. The reasoning trace answers, imperfectly, why the agent thought any of this was the right move. The audit trail is the durable assembly of all three, structured so that someone who was not present can reconstruct the session weeks later. None of these are interchangeable. A clean diff with no tool-call log hides whether the agent silently weakened a test on the way to the green checkmark. A reasoning trace without a diff tells you a story divorced from what actually shipped. You need all four, and you need to read them differently than you would read the equivalent artifacts from a human collaborator.

Reading agent diffs in particular requires habits that human-code review does not train you for. A human's diff usually expresses a single intent; an agent's diff often expresses an intent plus the residue of two or three abandoned approaches it didn't fully revert. Patterns that signal low-confidence generation — overly defensive try/except blocks wrapping things that don't throw, comments restating what the code already says, suspiciously narrow conditionals that handle exactly the test inputs and nothing more, helper functions used once — are noise in human code but are tells in agent code. The skill is recognising when a diff is solving the stated problem versus when it is solving the appearance of the stated problem, and that recognition does not transfer automatically from years of reviewing humans.

Reasoning traces deserve a specific warning. The chain-of-thought an agent emits is not a faithful account of why it acted; it is a plausible-sounding narrative generated alongside the actions, and it can be confidently wrong about its own causes. Treating the trace as ground truth is a category error. Treating it as one signal among several — useful for spotting confusion, useless for proving correctness — is the correct stance. When the trace and the tool-call log disagree, the tool-call log wins, because tool calls actually happened and reasoning is a story about them.

There is also a legibility problem one level up, in how the field talks about agents at all. Published benchmark scores — SWE-bench Verified is the canonical example — are reported as if they measured model capability, but they measure a model and a scaffold and a feedback environment together. A score is not legible if you cannot decompose it into those parts. The practical consequence is that you cannot import other people's evaluations to make decisions about your own codebase; you have to build a small, boring, repeatable evaluation against your own repository, your own scaffolds, your own tasks. That is not over-engineering. That is the only form of evaluation that produces a number you can actually trust.

The skill this topic builds is the discipline of refusing to grant autonomy that exceeds your visibility. Every time you extend an agent's permissions, the corresponding question is not "is the model good enough" but "will I be able to see what it did, in time to act on what I see." If the answer is no, the autonomy is not yet earned regardless of how capable the underlying model is. Observability is what converts speed into trustworthy speed; without it, the agent is producing artifacts faster than you can take responsibility for them, and you are accumulating ownership debt against code you cannot defend. The agent's job is to do the work. Your job, increasingly, is to remain in a position to know whether the work was done.

## Level 2 candidates

**Diff-level legibility** — Covers the specific patterns that distinguish agent-generated diffs from human-generated ones and the review habits required to catch low-confidence generation, abandoned approaches, and tests-passing-by-shape rather than substance. Worth a deep dive because the review skill is non-obvious, doesn't transfer from human code review, and is where most "the agent shipped bad code" failures are actually catchable.

**Reasoning traces and their limits** — Covers what chain-of-thought output is mechanically (a generated narrative, not a causal record), where it provides genuine signal, and the specific ways it misleads when treated as ground truth. Worth deeper treatment because the gap between how traces feel (transparent insight into the agent's mind) and what they are (post-hoc plausible text) is the source of a lot of misplaced trust.

**Audit trail design** — Covers what to log about agent sessions — tool calls, file reads, command output, model versions, scaffold configuration, timestamps — so that a session can be reconstructed by someone who wasn't there. Worth its own treatment because the minimum viable record is not obvious, and most teams discover what they should have logged only after an incident in which they didn't.

**Benchmark legibility and building your own evals** — Covers why published scores like SWE-bench Verified confound model, scaffold, and environment, and how to construct a small repeatable evaluation against your own codebase. Worth a deep dive because the gap between marketing benchmarks and decisions you actually need to make is wide, and the practical recipe for closing it is concrete enough to teach.

**The autonomy–legibility coupling** — Covers the specific operational rule that autonomy should never be extended past the point where you can still reconstruct what the agent did on a timescale that matches the speed at which it acted. Worth deeper treatment because it generalises into a calibration heuristic that touches L1-07 (blast radius) and L1-09 (the autonomy dial) and gives a concrete test for whether a workflow is over-trusted.
