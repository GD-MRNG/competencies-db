# L1-03 · The Feedback Environment

The reason coding agents work at all — and the reason they work much better at code than at almost anything else you might want to delegate to them — is not the model. It is that code is one of the few domains where a machine can ask "did that work?" and get an honest answer in seconds. Compilers, type checkers, linters, and test runners produce precise, machine-readable verdicts on whether a piece of work is correct. Strip that away and the agent is left guessing, with nothing to push back against its own confident-sounding output. The feedback environment — the set of checks the agent can actually run against its own work — is the structural reason the loop converges instead of drifting.

Once you see this, a lot of folklore about agent quality reorganises itself. People reach for better prompts and stronger models because those are the visible levers. The invisible lever, almost always more powerful, is what the environment will tell the agent in response to its actions. An agent operating in a repository with a fast test suite, strict types, and a working build is a different creature from the same agent operating in a repository where the only signal is "the file saved." The first can self-correct; the second is reduced to producing plausible text and hoping. Same model, same scaffold, radically different reliability — because the loop has something to close against, or it doesn't.

The distinction that matters here is between syntactic and semantic feedback. Syntactic feedback tells the agent that something happened: a command exited, a file was written, a process ran without crashing. Semantic feedback tells the agent whether what happened was correct, and ideally why it wasn't. A failing assertion with a stack trace is semantic — it names the gap between intent and behaviour. A zero exit code from a script that doesn't actually verify anything is syntactic theatre, and an agent given only that signal will happily declare victory on broken code. The richer and more semantic the feedback, the more iterations the agent can spend usefully; the thinner the feedback, the faster the loop degenerates into confabulation that sounds like progress.

This reframes a lot of "agent setup" work as feedback engineering. Making the test suite fast and runnable from inside the agent's environment is not a quality-of-life improvement; it is a capability upgrade for the agent itself. The same goes for type checks, linters, integration smoke tests, and any other check that produces a precise verdict the agent can read and react to. When you find yourself debugging why an agent keeps producing subtly wrong output, the first question is rarely "is the model good enough" and almost always "what could the agent have run to know it was wrong, and did it have access to that?" The leverage of a thirty-second test loop the agent can invoke autonomously is hard to overstate, and it is almost always worth the engineering cost to build.

The corollary is uncomfortable: agents are weakest exactly where you most want help. Tasks with no automatic ground truth — judgment calls, taste, ambiguous requirements, refactors whose correctness lives in someone's head, design decisions that won't fail any test — are precisely the ones where the loop has nothing to close against. The agent will still produce confident output on these tasks, because producing confident output is what models do. But the self-correction mechanism is gone, and what comes back is plausibility, not verification. Recognising this category before you delegate is the difference between using an agent well and being surprised by it. If you cannot articulate how the agent would know it had succeeded, the agent cannot know either.

Even when feedback is rich, the loop needs a budget. An ungrounded or partially grounded loop — one where the checks aren't quite catching the real failure — does not gracefully fail. It either burns tokens and tool calls in an indefinite retry spiral, or it converges on something that satisfies the visible signal while missing the underlying intent. Explicit limits on iterations, time, and cost are not just operational hygiene; they are an admission that not every task has a clean termination condition, and the loop needs an external one when the internal one is unreliable. This is where the feedback environment connects back to the rest of the system: the budget is what prevents a degraded environment from turning into a runaway one.

The skill this topic builds is a habit of mind. Before you give an agent a task, ask: how will it know when it's done? What can it run, and what will the output of that run actually tell it? If the answer is a real, semantic, fast-running check, the task is probably delegable and the agent is probably going to get there. If the answer is "I'll look at it when it's finished," you haven't given the agent a feedback environment — you've given it a generation prompt with extra steps, and the reliability you'll get reflects that. Engineer the loop the agent runs inside, and most of the rest of agent quality follows.
